## 一、数据持久化/状态一致性

### `BusinessException` 回滚吞掉写入操作

多处代码通过抛出 `BusinessException` 回滚事务，覆盖了同一事务内的必需写入：

| 文件 | 行号 | 场景 |
|------|------|------|
| `DungeonService.java:322` | 战斗失败先 `markFailed()` → save → 再 throw → 回滚 → 失败标记丢失 |
| `DungeonService.java:293` | `checkExpired` 先 `markAbandoned()` → save → 再 throw → 过期标记丢失 |
| `PostCombatProcessor` | HP写入在事务内，但调用方可能因异常回滚 |

### 多重保存导致的潜在数据覆盖

**文件:** `TrainingCombatLogic.java:200-202` + `TrainingService.java:176-199`  
**触发:** `TrainingCombatLogic.finalizeTraining` 保存了含战斗经验的 user 对象，同一事务内 `TrainingService.processNormalTrainingEnd` 重新 `loadUser`（可能读到未刷新的旧数据）→ `addExp` → `save` — 若缓存未刷新，第二次 save 覆盖战斗经验。

### `buildCell` 建设中覆盖升级过的地块等级

**文件:** `FudiService.java:369`  
**触发:** 曾升级到Lv3的地块若被拆除重建，`cell.setCellLevel(1)` 将等级重置 — 升级投资完全消失。

### 配方卷轴使用后在 `learnRecipe` 异常时丢失

**文件:** `RecipeScrollUseHandler.java:24-29` + `ItemUseService.java:70-80`  
**触发:** `ItemUseService.useItem` 无 `@Transactional`；scroll 在 `reduceStackableItem`（自带事务）中被扣除后提交，若 `pillRecipeService.learnRecipe` 抛出 `RECIPE_ALREADY_LEARNED` 则卷轴已消失但配方未学到。

### 使用物品时扣除与使用不在同一事务

**文件:** `ItemUseService.java:35-81`  
**触发:** `reduceStackableItem` 自有 `@Transactional` 并提交，若后续 `handler.use()` 抛异常（如血量满无法吃药），物品已被扣除但效果未生效。

### 收获作物时模板缺失静默归零

**文件:** `FarmService.java:205-220`  
**触发:** `grantHarvestItems` 中种子模板为 null 或非 Growth 类型时 `return`，作物被收获但产量为0，无任何报错。

## 二、并发竞态

### 突破系统双重突破竞态

**文件:** `CultivationService.java:85-111`  
**触发:** 无 `SELECT FOR UPDATE`、无 `@Version`、无乐观锁。两个并发突破请求均读同一 `user.level/exp`，均通过检查，均升级 — 玩家获得免费直升。

### 同一装备槽同时装备两件物品

**文件:** `EquipmentService.java:67-69,104-109`  
**触发:** 竞态条件下两并发 `equipItem` 请求同时读取空槽位，两件装备均设为 `equipped=true`。

### 叠加物品重复堆栈

**文件:** `StackableItemService.java:40-55`  
**触发:** 两并发 `addStackableItem` 均发现 `existingItem.isEmpty()`，均创建新行 → 重复堆栈。缺少唯一约束+UPSERT。

### 守护者数量限制竞态

**文件:** `DaoProtectionService.java:72-86`  
**触发:** `countByProtectorId` 读后 `save` 写入之间存在竞态窗口，可超越 `MAX_PROTECTOR_COUNT` 限制。

### 用户状态检查-后-设置竞态

| 文件 | 场景 |
|------|------|
| `BountyService.java:138,170` | IDLE → BOUNTY |
| `DungeonService.java:138,159` | IDLE → DUNGEON |
| `TrainingService.java:66,92` | IDLE → TRAINING |
| `TravelService.java:38-40,70-74` | IDLE → TRAVELING |

均使用 `READ_COMMITTED` 隔离级别下检查后修改，无行锁。

### 代码仓缺少任何并发保护手段

- **无** `SELECT ... FOR UPDATE`  
- **无** `@Version` 乐观锁  
- **无** `synchronized` 块  
- **仅3处**原子SQL更新（`deductSpiritStonesIfEnough`、`tryClaimDailyGift`、`upsertIncrementCount`）

## 三、战斗系统

### 攻速不影响技能冷却恢复

**文件:** `DefaultCombatEngine.java:188,378`  
**触发:** `Combatant.getAttackSpeed()` 存在（默认1.0）但冷却倒计数固定每回合-1。攻速属性对战技能的角色无意义。

### `FREEZE` 描述"受伤增加"但伤害放大未实现

**文件:** `DefaultCombatEngine.java` — `BuffType.FREEZE` 描述声明了额外受伤但从未应用。

### 据点场景被忽略 — PvP与PvE目标选择逻辑完全相同

**文件:** `DefaultCombatEngine.java:374-376`  
`BattleContext.scene` 从未用于区分目标选择策略。

### DOT伤害过度

**文件:** `DefaultCombatEngine.java:327`  
3层叠加时 `attacker.atk * 0.3 * 3` 每回合造成巨额被动伤害，结合刷新冠军可支配战斗。

### `Battle.execute()` 非幂等

**文件:** `Battle.java:55-61`  
重复调用会使用已变异的 `Combatant` (HP已归零) 重新模拟，产出垃圾结果。`executed` 字段设了但从未读取。

### 历练战斗遇敌间无HP恢复

**文件:** `TrainingCombatLogic.java:153-178`  
12次连续遇敌累计伤害无恢复 → 长时间历练必然死亡。

### 历练敏捷效率倍率无上限（爆炸性增长）

**文件:** `TrainingService.java:288`  
`1.0 + agility * 0.01`，500敏捷可产出正常6倍经验。结合无HP恢复 → 高敏捷必死。

## 四、核心经济/养成

### 丹药Buff无堆叠上限

**文件:** `PillConsumptionService.java:189-208`  
可无限堆叠同类型Buff（ATTACK/DEFENSE/SPEED/BREAKTHROUGH）。20个攻击Buff全部生效 → 轻易突破战斗平衡。

### 丹药品质共享同一抗性计数

**文件:** `PillConsumptionService.java:110,171`  
抗性键为 `templateId`。吃下品丹增加的抗性同样降低上品丹效果。玩家被惩罚尝试低品质丹药。

### 默认 `expiresAt=null` 让Buff永久化

**文件:** `PlayerBuff.java:38-40`  
未设置过期时间的Buff永远不会过期 — 可意外获得永久属性加成。

### 丹药HP百分比疑似Bug

**文件:** `PillConsumptionService.java:123`  
`e.percentage() / 100 * qualityMultiplier` — 若 `percentage` 是百分比（如30代表30%）可正常工作，若为小数（0.3）则 `0.3/100=0.003` ≈ 0。

### 锻造 `statBonus` 总是包含零值属性

**文件:** `ForgingService.java:191-196`  
锻造装备stats始终包含全部4属性 (STR/CON/AGI/WIS)，即使为0。而商店装备则过滤零值 — 不一致可能破坏显示或统计。

### 商店讨价还价无限利用

**文件:** `ShopService.java:355-377`  
无每日限制、无冷却、`currentPrice` 调用者自供。玩家可通过反复还价以5%复合折扣将价格压至极低。

### 商店卖出任意报价接受无谈判

**文件:** `ShopService.java:228`  
卖出时玩家可永远设定 `confirmedPrice = maxPrice` 获得最高价格，无失败风险。卖价系统成为无操作按钮。

### 店铺补货永久上限为5

**文件:** `PriceEngine.java:142-143`  
售罄物品补货量封顶为5，无视 `maxStock`。热门物品永远停留在 0→5→0 循环。

### `(int)` 强转灵石价格溢出

**文件:** `ShopService.java:119,160`  
`User.spiritStones` 是 `Long` 但付款接口用 `(int) price` 强转 — 单项超21亿价格溢出，玩家可能付极少代价。

## 八、代码质量 / 维护性

### 自调用使20+处 `@Transactional` 失效（误导性标注）

| 表现: |
|------|
| `ForgingService.forgeAuto(Long)` ← `forgeAuto(PlatformType,...)` |
| `EnhancementService.enhanceAuto(Long)` ← `enhanceAuto(PlatformType,...)` |
| `CultivationService.attemptBreakthrough(Long)` ← `attemptBreakthrough(PlatformType,...)` |
| `ShopService.purchaseItem(Long)` ← `purchaseItem(PlatformType,...)` |

外部调用→外部`@Transactional` 有效，内部 `@Transactional` 被 Spring 代理绕过（`this.xxx()` 不走代理）。标注误导维护者。

### 子串 `name.contains()` 匹配导致非确定性物品定位 — 13处

| 文件 | 方法 |
|------|------|
| `PillRefiningService:60,174,178` | 药材和目标匹配 |
| `PillConsumptionService:246-248` | 丹药名称匹配 |
| `PillRecipeService:78` | 配方名称匹配 |
| `ItemUseService:37-49` | 物品名称匹配 |
| `ForgingService:87,251,267,360` | 图纸/材料匹配 |
| `FarmService:326-331` | 种子名称匹配（`.findFirst()` 无序） |
| `BeastBreedingService:88-92` | 兽卵名称匹配（`.findFirst()` 无序） |

应统一使用 `ItemResolver` 的精确匹配优先策略。

### `Math.random()` 全局竞争替代 `ThreadLocalRandom`

**文件:** `EquipmentService.java:226`  
`Math.random()` 使用全局 `AtomicLong` 在高并发下导致CAS自旋竞争。`ThreadLocalRandom` 已import但在此处未使用。

### `CombatService` 重复buff/武器加载逻辑

**文件:** `CombatService.java:44-58 ≡ 96-110`  
两个 `buildPlayerTeam` 重载方法完全重复buff循环和灵兽循环。修改需同步两处。

### 伤害计算器字符串替换脆弱

**文件:** `DamageCalculator.java:80-86`  
`.replace("atk", ...)` 会错误匹配未来可能添加的 `atk_bonus` 变量（先匹配"atk"生成废话）。应使用正则 `\b` 边界。

### 自定义表达式解析器无输入验证

**文件:** `DamageCalculator.java:106-180`  
处理 `+`/`-`/`*`/`/`/`(`/`)`/整数但无溢出检测、无零除保护（仅靠catch兜底），无负数支持（除减法结果的中间值外）。

### NPC性格检查匹配任意含"T"字符串

**文件:** `ShopService.java:359`  
`npc.getPersonality().contains("T")` 匹配"HONEST"、"TRADER"、"TALKATIVE"等非预期结果。

### 数值硬编码散布各处

| 魔术数字 | 文件 | 含义 |
|----------|------|------|
| `2` | `CombatService.java:82` | 灵兽最大出战数 |
| `3` | `CharacterStatusService.java:107` | 守护者最大显示数 |
| `1` | `FudiGiftService.java:44` | 模板ID回退值 |

### `PillRecipeService.getRecipeDetail` 返回 `Success<(>null)`

**文件:** `PillRecipeService.java:73-87`  
找不到配方时返回 null 而非抛异常 — 调用方收到"成功"但数据为空。

### 练功中技能觉醒使用 `skillRepository.findAll()`

**文件:** `TrainingCombatLogic.java:482`  
每次觉醒立即加载全部技能进内存过滤。无缓存、无DB分页。大量技能时性能线性退化。
