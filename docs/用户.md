### 1.1 核心字段

| `stat_wis`                    | INT              | 悟性                  |

| `location_id`                 | BIGINT           | 所在地图ID              |
| `activity_type`               | VARCHAR(16)      | 通用活动类型 (TRAVEL/TRAINING/BOUNTY) |

### 1.2 索引

## 3. 等级与经验

### 3.1 升级公式

```
升级所需经验 = 100 × level²
经验存储上限 = 升级所需经验 × 5
```

### 3.2 突破效果

**成功**:

- level += 1
- 扣除升级所需经验 (exp -= expNeeded)
- 全属性实际值 += 1（基础值 = 4 + level，由等级推导）
- 失败计数归零
- HP 回满
- 清除所有被护道关系
- 清除所有突破类 Buff

**失败**:

- 扣除全部所需突破修为 (exp = max(0, exp - expNeeded))
- 失败计数 += 1
- 清除所有被护道关系
- 清除所有突破类 Buff

## 4. 四维属性

### 4.1 基础值

```
基础属性 = 4 + level
实际属性 = 基础属性 + 丹药加成 + 装备加成

等级 1: 全属性 = 5（无加成时）
等级 50: 全属性 = 54（无加成时）
每升一级：全属性 +1
```

丹药（Stat 类）可直接叠加到指定属性字段上（`stat_str/con/agi/wis`）。

### 4.2 属性公式

| 属性       | 影响                                                       |
|----------|----------------------------------------------------------|
| 力量 (STR) | 法器普攻伤害 `attack = str × 2 + equipAttack`                  |
| 体质 (CON) | HP上限 `100 + con × 20`，防御值 `defense = con + equipDefense` |
| 敏捷 (AGI) | 战斗效率（增加攻击轮次）                                             |
| 智慧 (WIS) | 法决伤害 `wis × scale + base`                                |

### 4.3 差异化来源

玩家间的属性差异**不来自加点**，而来自：

- **装备**：每件装备有不同的 `base_str/con/agi/wis` 加成 + 随机词条
- **法器类型**：不同法器对怪物有不同克制倍率
- **法决**：智慧高 → 法决伤害高

## 5. 突破系统

### 5.1 成功率公式

```
总成功率 = min(100, 基础概率 + 失败次数 × 单次补偿 + 护道加成 + 突破丹药加成)
```

- 突破丹药加成：来自 `xt_player_buff` 表中未过期的 `buff_type='breakthrough'` 记录，取总和的 value
- 突破后（无论成败）清除该玩家所有 `breakthrough` 类 Buff
- `attemptBreakthrough()` **不检查玩家当前状态**，任何状态下均可发起突破

### 5.2 护道加成

```
单人加成 = 5% + (护道者等级 - 被护道者等级) × 1%
总加成上限 = 20%（仅同地点护道者生效）
```

## 7. 用户状态 (UserStatus)

| 状态         | 说明            |
|------------|---------------|
| IDLE       | 空闲            |
| TRAINING   | 历练中           |
| TRAVELING  | 旅途中           |
| BOUNTY     | 悬赏任务中         |
| DYING      | 濒死（HP=1，无法行动） |

### 7.1 濒死 (DYING)

触发条件：战斗中 HP 归零后调用 `setDying()`（HP 固定设为 1）。

效果：

- 无法进行任何行动（历练/旅行/悬赏）
- 可使用丹药恢复 HP，HP > 0 后状态回到 IDLE
- 或等待 30 分钟自动恢复（见 7.3）

### 7.2 HP 自然恢复

```
IDLE 状态：每5分钟恢复 1% 最大HP，直到满血
其他状态：不恢复
```

**实现机制**（懒加载）：

- 通过 `last_hp_recovery_time` 列记录上次恢复时间戳
- `UserStateService.tryHpRecovery()` 计算经过的 5 分钟刻度数，批量执行 `user.naturalHpRecovery()`
- 首次加载时 `last_hp_recovery_time` 为空 → 写入当前时间为初始值，不触发恢复
- 每个刻度恢复 `max(1, maxHp / 100)` 点 HP
- 时间戳按刻度对齐推进（而非直接设为当前时间），避免累计误差

```
resolveState(user)
  └─ tryHpRecovery(user)
       ├── status != IDLE → skip
       ├── hpCurrent >= maxHp → skip
       ├── lastHpRecoveryTime == null → 写入当前时间，return true
       ├── ticks = elapsedMinutes / 5
       ├── ticks <= 0 → skip
       ├── for tick in ticks: user.naturalHpRecovery()
       └── 更新 lastHpRecoveryTime = lastRecovery + ticks × 5

HP 满时产出 HP_RECOVERED 事件到 xt_game_event，由 NotificationAppender 追加到回复。

### 7.3 濒死自动恢复

玩家处于 DYING 状态超过 30 分钟后自动恢复：

- HP 恢复到 20% 最大HP（最低 1 点）
- 状态回到 IDLE
- `activity_start_time` 清空
- 追踪字段：`dying_start_time`（TIMESTAMP）
- 产出 DYING_RECOVERED 事件到 xt_game_event

```
resolveState(user)
  └─ tryDyingRecovery(user)
       ├── status != DYING → skip
       ├── dyingStartTime == null → 写入当前时间作为起始点，return true
       ├── elapsedMinutes < 30 → skip（恢复期未到）
       ├── 恢复 HP 到 max(1, maxHp/5)
       ├── status → IDLE
       ├── dyingStartTime → null
       └── activity 清空
```

### 7.4 旅行自动结算

旅行到期后在下一次用户交互时自动结算，详见 [地图旅行历练.md](./地图旅行历练.md#42-旅行自动结算)。

规则与提升的懒加载机制一致——状态查询/操作时即时计算。

## 8. 用户状态服务 (UserStateService)

### 8.1 设计动机

所有服务加载用户时都需要确保状态一致（旅行是否到期、HP 是否恢复、DYING 是否超时、buff 是否过期）。之前这些逻辑散落在多个调用点，容易遗漏且难以扩展。

`UserStateService` 作为统一入口，所有 `userRepository.findById(userId).orElseThrow()` 替换为 `userStateService.loadUser(userId)`：

`UserStateService` 是一个 `@Service` 组件，依赖 `UserRepository`、`MapNodeRepository`、`PlayerBuffRepository`。

**公开方法：**
- `loadUser(Long userId)` → User：根据 ID 加载用户，调用 `resolveState()` 确保状态一致后返回
- `save(User)` → User：持久化用户状态

**内部方法：**
- `resolveState(user)`：依次尝试旅行结算、濒死恢复、HP 自然恢复、过期 Buff 清理，若任一操作导致数据变化则持久化

**常量：**
- HP 恢复间隔：5 分钟
- 濒死超时：30 分钟
- `extraData` 键：`lastHpRecovery`、`dyingSince`

### 8.2 扩展点

后续需要新增自动状态解析时，只需在 `resolveState()` 中追加一个 `try*` 方法：

| 扩展方法 | 描述 |
|---------|------|
| `tryCompleteTravel()` | 旅行到期结算 |
| `tryDyingRecovery()` | 濒死超时自动恢复（30分钟） |
| `tryHpRecovery()` | HP 自然恢复（extraData 追踪） |
| `tryExpireBuffs()` | 过期 buff 清理（单条 DELETE） |
| *(未来)* `tryResolveCombat()` | 战斗超时处理 |
| *(未来)* `tryRegenMana()` | 灵力自然恢复 |

### 8.3 迁移范围

13 个服务文件、27 处调用点统一迁移至 `userStateService.loadUser(userId)`：

| 服务 | 变更 |
|------|------|
| BountyService | userRepository → userStateService |
| CharacterStatusService | userRepository → userStateService |
| CultivationService | userRepository → userStateService（保留 UserRepository 用于 findByNickname 等） |
| EquipmentService | userRepository → userStateService |
| FudiHelper | userRepository → userStateService |
| InventoryService | userRepository → userStateService |
| MapService | userRepository → userStateService |
| PillConsumptionService | userRepository → userStateService |
| SkillService | userRepository → userStateService |
| StackableItemService | userRepository → userStateService |
| TrainingService | userRepository → userStateService |
| TravelService | userRepository → userStateService |
| — | UserStateService（新增） |

`UserStateService` 提供 `loadUser()` 和 `save()` 两个公开方法。
