# 用户系统设计文档

## 1. 模块概述

用户系统是仙道游戏的核心基础模块，负责管理玩家角色信息、跨平台身份认证、属性成长体系等核心功能。

### 1.1 核心职责

- 角色创建与信息管理
- 跨平台账号绑定（QQ、Web等）
- 四维属性体系（力量、体质、敏捷、智慧）
- 等级与经验管理
- 突破与洗点机制
- 护道关系管理

**设计意图**：允许一个游戏角色绑定多个不同平台的账号，但同一平台账号只能绑定一个角色。

## 3. 核心业务规则

### 3.1 角色创建流程

#### 3.1.1 注册流程

1. 验证平台OpenID是否已绑定角色
2. 检查道号唯一性
3. 创建角色记录（初始属性均为5，等级1，经验0）
4. 创建跨平台授权记录
5. 返回角色ID和道号

#### 3.1.2 初始化配置

- 初始等级：1
- 初始经验：0
- 初始属性：力量5、体质5、敏捷5、智慧5
- 初始生命值：200（100 + 5×20）
- 初始体力值：100
- 可用属性点：0

### 3.2 等级与经验系统

#### 3.2.1 升级公式

```
升级到下一级所需经验 = 100 × 当前等级²
```

**示例**：

- 1级升2级：100 × 1² = 100经验
- 2级升3级：100 × 2² = 400经验
- 10级升11级：100 × 10² = 10000经验

#### 3.2.2 经验存储上限

```
最大存储经验 = 升级所需经验 × 3
```

**设计意图**：防止玩家长期离线后积累过多经验，保持游戏节奏。

**示例**：

- 10级玩家升级需10000经验，最多存储30000经验
- 超出部分无法获取，提示"修为已达瓶颈"

#### 3.2.3 升级流程

1. 检查当前经验 ≥ 升级所需经验
2. 扣除升级所需经验（exp -= 升级经验）
3. 等级+1
4. 获得1点自由属性点（free_stat_points++）
5. 生命值恢复至满值
6. 重置突破失败计数为0
7. 清除该角色的所有被护道关系

### 3.3 四维属性体系

#### 3.3.1 属性定义

| 属性 | 英文缩写 | 影响范围              |
|----|------|-------------------|
| 力量 | STR  | 破坏力、锻造成功率、装备攻击力加成 |
| 体质 | CON  | 生命值上限、物理防御、体力上限   |
| 敏捷 | AGI  | 出手顺序、历练效率倍率、闪避率   |
| 智慧 | WIS  | 经验加成、探索掉宝率、炼药成功率  |

#### 3.3.2 属性计算公式

**最大生命值**：

```
HP上限 = 100 + 体质 × 20
```

**最大体力值**：

```
体力上限 = 100 + 体质 × 5
```

**历练效率倍率**：

```
效率倍率 = 1.0 + 敏捷 × 0.01
```

- 10点敏捷 → 1.1倍收益
- 50点敏捷 → 1.5倍收益
- 100点敏捷 → 2.0倍收益

**探索有效智慧**：

```
有效智慧 = 智慧 - (地图等级 - 1) × 5
最低有效智慧 = 1
```

#### 3.3.3 属性分配规则

- 每次升级获得1点自由属性点
- 可一次性分配多点（如5点全加力量）
- 分配后立即生效，无需确认

### 3.4 突破系统

#### 3.4.1 突破成功率算法

突破成功率随等级提升呈S型曲线下降，通过失败补偿机制保证长期可玩性。

**基础概率公式**：

```
基础概率 = 100 / (1 + (等级 / 50)^4.5)
```

**失败补偿增量**：

```
单次补偿 = 3 + 17 / (1 + (等级 / 50)²)
总成功率 = 基础概率 + (失败次数 × 单次补偿)
```

**范围限制**：最终成功率限制在 [0%, 100%] 区间内。

**参数说明**：

- **baseMidpoint = 50**：基础概率减半的等级点
- **baseSteepness = 4.5**：概率下降的陡峭度
- **minPity = 3**：最小失败补偿（3%）
- **maxPityOffset = 17**：额外最大补偿（总计20%）

#### 3.4.2 成功率曲线示例

| 等级  | 基础概率  | 失败1次  | 失败5次  | 失败10次 |
|-----|-------|-------|-------|-------|
| 10  | 99.9% | 100%  | 100%  | 100%  |
| 30  | 98.5% | 100%  | 100%  | 100%  |
| 50  | 50.0% | 56.5% | 82.5% | 100%  |
| 70  | 12.3% | 17.8% | 38.3% | 64.3% |
| 100 | 1.2%  | 4.7%  | 16.7% | 32.2% |

#### 3.4.3 突破结果

**成功**：

- 等级+1
- 经验扣除（exp -= 升级所需经验）
- 获得1点自由属性点
- 突破失败计数归零
- 生命值恢复至满值
- **清除该角色的所有被护道关系**

**失败**：

- 扣除当前境界升至下境界所需的标准修为值
- 突破失败计数+1
- **清除该角色的所有被护道关系**

**设计意图**：突破失败惩罚较重，鼓励玩家寻求护道者帮助。

### 3.5 洗点系统

#### 3.5.1 洗点规则

- 将所有已分配的属性点重置为基础值（5点）
- 返还的点数加入可用属性点池
- 冷却时间：72小时（3天）

#### 3.5.2 洗点计算公式

```
总已分配点数 = (力量-5) + (体质-5) + (敏捷-5) + (智慧-5)
返还点数 = 总已分配点数
重置后各属性 = 5
可用属性点 += 返还点数
```

#### 3.5.3 冷却判定

```
可洗点条件 = (当前时间 - 上次洗点时间) ≥ 72小时
首次洗点无冷却
```

### 3.6 护道系统

#### 3.6.1 护道加成计算

**单人加成公式**：

```
单人加成 = 5% + (护道者等级 - 被护道者等级) × 1%
```

**总加成计算**：

- 仅统计与被护道者在**同一地点**的护道者
- 累加所有同地点护道者的单人加成
- 总加成上限：**20%**

**示例**：

- 护道者50级，被护道者40级 → 单人加成 = 5% + 10×1% = 15%
- 若有2个同地点护道者，总加成 = 15% + 15% = 30%，但受上限限制为20%

#### 3.6.2 护道约束

**建立护道关系**：

1. 护道者境界必须 ≥ 被护道者境界
2. 护道者当前护道人数 < 3
3. 不存在重复护道关系

**解除护道关系**：

- 护道者可主动解除
- 被护道者突破后自动清除所有护道关系

**位置判定**：

- 检查护道者与被护道者的location_id是否相同
- 只有同地点才提供加成

#### 3.6.3 护道查询

**双向查询能力**：

- **我护道的人**：显示被护道者信息、是否同地点、提供的加成
- **护道我的人**：显示护道者信息、是否同地点、实际生效的加成

## 4. 服务层设计

### 4.1 UserService

**职责**：用户创建、基础信息查询

**核心方法**：

- `createUser(platform, openId, nickname)`：创建新用户
- `getUserById(userId)`：根据ID查询用户
- `findByNickname(nickname)`：根据道号查询用户

### 4.2 UserAuthService

**职责**：跨平台身份认证

**核心方法**：

- `authenticate(platform, openId)`：验证平台身份，返回userId
- `bindPlatform(userId, platform, openId)`：绑定新平台
- `unbindPlatform(platform, openId)`：解绑平台

### 4.3 CultivationService

**职责**：修仙核心逻辑（加点、洗点、突破、护道）

**核心方法**：

#### 4.3.1 属性管理

- `allocateAttributePoints(userId, attributeType, points)`：分配属性点
- `resetStatPoints(userId)`：洗点（重置所有属性）

#### 4.3.2 突破管理

- `attemptBreakthrough(userId)`：尝试突破

#### 4.3.3 护道管理

- `establishProtection(protectorId, protegeNickname)`：建立护道关系
- `removeProtection(protectorId, protegeNickname)`：解除护道关系
- `queryProtectionInfo(userId)`：查询护道信息

### 4.4 StaminaService

**职责**：体力管理

**核心方法**：

- `getStaminaInfo(userId)`：获取体力信息（含离线恢复计算）
- `checkAndConsumeTrainingStamina(userId, durationMinutes)`：检查并消耗历练体力
- `checkAndConsumeExplorationStamina(userId)`：检查并消耗探索体力
- `restoreStaminaByMeditation(userId, durationMinutes)`：打坐恢复体力
- `restoreStaminaByItem(userId, staminaAmount)`：使用丹药恢复体力

## 5. 体力系统设计

### 5.1 体力恢复机制（懒加载）

#### 5.1.1 恢复速率

- **空闲状态(IDLE)**：每5分钟恢复1点体力
- **打坐状态(MEDITATING)**：每分钟恢复2点体力
- **其他状态**：不恢复

#### 5.1.2 离线恢复计算

系统不主动更新体力值，而是在玩家查询或执行操作时即时计算。

**计算公式**：

```
经过分钟数 = (当前时间 - lastStaminaUpdateTime) 的分钟差

if (状态 == MEDITATING):
    恢复量 = 经过分钟数 × 2
else if (状态 == IDLE):
    恢复量 = 经过分钟数 / 5  // 整除
else:
    恢复量 = 0

实际恢复量 = min(恢复量, 最大体力 - 当前体力)
lastStaminaUpdateTime = 当前时间
```

**优势**：

- 减少数据库写入频率
- 支持真正的离线计算
- 简化并发控制

### 5.2 体力消耗场景

| 操作       | 消耗规则        |
|----------|-------------|
| 历练       | 每分钟消耗1点体力   |
| 探索       | 每次固定消耗20点体力 |
| 旅行（体力模式） | 每分钟消耗5点体力   |
| 打坐       | 不消耗体力，反而恢复  |

### 5.3 体力不足处理

- 历练/探索前检查体力是否足够
- 不足时返回错误提示，引导玩家打坐或使用丹药
- 不扣减体力，操作不执行

## 6. 跨平台命令处理

### 6.1 命令处理器 (CultivationCommandHandler)

**职责**：统一处理跨平台的修仙相关命令

**核心方法**：

#### 6.1.1 用户管理

- `handleRegister(platform, openId, nickname)`：处理注册命令
- `handleStatus(platform, openId)`：处理状态查询

#### 6.1.2 属性管理

- `handleAllocatePoints(platform, openId, attributeName, points)`：处理属性加点
- `handleResetPoints(platform, openId)`：处理洗点

#### 6.1.3 突破与护道

- `handleBreakthrough(platform, openId)`：处理突破命令
- `handleEstablishProtection(platform, openId, protegeNickname)`：处理护道建立
- `handleRemoveProtection(platform, openId, protegeNickname)`：处理护道解除
- `handleQueryProtection(platform, openId)`：处理护道查询

### 6.2 身份认证流程

```
平台监听器接收消息
    ↓
调用 authenticate(platform, openId)
    ↓
查询 xt_user_auth 表
    ↓
若存在 → 返回 userId
若不存在 → 返回 "未注册，请输入「我要修仙 道号」"
    ↓
CommandHandler 执行业务逻辑
    ↓
格式化响应消息
    ↓
返回给平台监听器渲染
```

## 7. 安全性与校验

### 7.1 输入校验

#### 7.1.1 属性分配校验

- 点数必须 > 0
- 可用属性点 ≥ 请求分配的点数
- 属性类型必须合法（STR/CON/AGI/WIS）

#### 7.1.2 突破校验

- 当前经验 ≥ 升级所需经验
- 用户存在且状态正常

#### 7.1.3 护道校验

- 护道者境界 ≥ 被护道者境界
- 护道者当前护道人数 < 3
- 被护道者存在
- 不存在重复护道关系

### 7.2 资源限制

#### 7.2.1 护道上限

- 单个护道者最多为3人护道
- 防止滥用护道加成

#### 7.2.2 洗点冷却

- 72小时冷却时间
- 防止频繁重置属性点破坏游戏平衡

### 7.3 数据一致性

#### 7.3.1 事务管理

- Service层方法添加@Transactional注解
- 确保属性分配、突破等操作原子性

#### 7.3.2 并发控制

- 使用Virtual Threads处理高并发请求
- 避免ThreadLocal，使用UserContext传递上下文
- 在finally块中清理上下文

## 8. 性能优化

### 8.1 数据库优化

#### 8.1.1 索引策略

- 排行榜字段：level、coins、spirit_stones（降序索引）
- 外键字段：user_id、platform_open_id
- 唯一约束：nickname、(platform, platform_open_id)

#### 8.1.2 查询优化

- 批量查询护道关系（避免N+1问题）
- 合理使用JOIN连接用户表和护道表

### 8.2 懒加载优势

**体力恢复**：

- 传统方案：每分钟定时任务更新所有在线玩家体力
- 懒加载方案：查询时计算，减少99%的写入操作

**历练收益**：

- 传统方案：每分钟累积收益到数据库
- 懒加载方案：结束时一次性计算，减少写入频率

## 9. 扩展性设计

### 9.1 新属性扩展

如需新增属性（如"悟性"）：

1. 在xt_user表添加stat_wuxing字段
2. 在AttributeType枚举添加WUXING
3. 在User实体添加getter/setter
4. 在相关业务逻辑中使用新属性

### 9.2 新平台接入

1. 在PlatformType枚举添加新平台类型
2. 创建新平台监听器（如handle/discord/）
3. 调用现有的CultivationCommandHandler
4. 无需修改核心业务代码

### 9.3 护道规则调整

- 修改MAX_PROTECTOR_COUNT常量即可调整护道上限
- 调整BASE_BONUS_PERCENTAGE和LEVEL_DIFF_BONUS_PERCENTAGE改变加成公式

## 附录B：枚举定义

### B.1 用户状态 (UserStatus)

- IDLE：空闲
