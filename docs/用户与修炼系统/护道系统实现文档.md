# 护道系统实现文档

## 概述

护道系统是 XianTao 的核心社交互助玩法，完全基于道友间的人情羁绊，无功德奖励，无系统收益。本文档详细记录护道系统的技术实现。

**实现状态**: ✅ 已完成

---

## 一、核心设计规则

### 1.1 基础约束

| 规则项         | 实现方式              | 状态    |
|-------------|-------------------|-------|
| **境界压制**    | 护道者境界 `>=` 突破者境界  | ✅ 已实现 |
| **护道者数量限制** | 最多同时为 **3名** 道友护道 | ✅ 已实现 |
| **突破者数量限制** | 接受无限多护道者          | ✅ 已实现 |
| **位置检测**    | 突破瞬间实时校验同地图节点     | ✅ 已实现 |

### 1.2 加成计算公式

```text
// 单人加成公式
单人加成 = 5% + (护道者境界层级 -突破者境界层级) × 1%

// 总加成公式
总加成 = min(20%,所有【同地点】护道者加成之和)
```

**实现位置**: `CultivationService.java`

- `calculateSingleProtectorBonus()` - 单人加成计算
- `calculateProtectionBonus()` - 总加成计算（含20%上限截断）

### 3.1 实体类

**文件**: `domain/user/entity/DaoProtection.java`

```java

@Table("xt_dao_protection")
```

## 四、核心业务逻辑

### 4.1 建立护道关系

**方法**: `CultivationService.establishProtection()`

**校验流程**:

```
1. 护道者存在性检查
   ↓
2. 被护道者存在性检查（按道号查找）
   ↓
3. 境界压制校验：protector.level >= protege.level
   ↓
4. 护道者名额检查：countByProtectorId < 3
   ↓
5. 重复关系检查：findByProtectorAndProtege
   ↓
6. 创建护道关系并保存
```

### 4.2 突破时护道加成计算

**方法**: `CultivationService.attemptBreakthrough()`

**执行流程**:

```
1. 修为充足性检查
   ↓
2. 计算护道加成（实时位置校验）
   - 查询所有护道者：findByProtegeId(userId)
   - 遍历检查是否同地点：isInSameLocation()
   - 累加同地点护道者的单人加成
   - 截断至20%上限
   ↓
3. 最终成功率 = min(100%, 基础成功率 + 护道加成)
   ↓
4. 随机判定突破成功/失败
   ↓
5. 清除所有护道关系（无论成功失败）
   ↓
6. 保存用户新状态
```

**位置校验逻辑**:

```java
private double calculateProtectionBonus(User protege) {
    List<DaoProtection> protections = daoProtectionRepository.findByProtegeId(protege.getId());
    double totalBonus = 0.0;

    for (DaoProtection protection : protections) {
        Optional<User> protectorOpt = userRepository.findById(protection.getProtectorId());
        if (protectorOpt.isPresent()) {
            User protector = protectorOpt.get();

            // 检查是否在同地点
            if (isInSameLocation(protector, protege)) {
                totalBonus += calculateSingleProtectorBonus(protector, protege);
            }
        }
    }

    return Math.min(MAX_TOTAL_BONUS_PERCENTAGE, totalBonus);
}

private boolean isInSameLocation(User user1, User user2) {
    if (user1.getLocationId() == null || user2.getLocationId() == null) {
        return false;
    }
    return user1.getLocationId().equals(user2.getLocationId());
}
```

### 4.3 突破后自动解除护道

**方法**: `CultivationService.clearProtegeRelations()`

```java
private void clearProtegeRelations(Long protegeId) {
    daoProtectionRepository.deleteByProtegeId(protegeId);
}
```

**设计决策**:

- ✅ 突破成功后自动解除（无论成功/失败）
- ✅ 不通知护道者（当前版本）
- ⚠️ 待优化：通过消息队列通知护道者突破结果

### 4.4 解除护道关系

**方法**: `CultivationService.removeProtection()`

**执行流程**:

```
1. 查找被护道者（按道号）
   ↓
2. 查找护道关系
   ↓
3. 删除护道关系
   ↓
4. 返回操作结果
```

### 4.5 护道查询

**方法**: `CultivationService.queryProtectionInfo()`

**查询内容**:

1. **作为护道者**: 显示正在为谁护道
    - 被护道者道号、境界、所在地图
    - 是否同地点标记
    - 单人加成百分比

2. **作为被护道者**: 显示谁在为自己护道
    - 护道者道号、境界、所在地图
    - 是否同地点标记
    - 实际提供加成（异地为0%）
    - 总加成（ capped at 20%）

**查询结果示例**:

```
共有 2 位道友为你护道，其中 1 位在同地点，总加成 6.0%

【你正在为以下道友护道】(1/3)
  李小白（第3层）- 青云山 [同地点] - 加成7.0%

【以下道友正在为你护道】总加成：6.0%
  张三丰（第5层）- 青云山 [同地点] - 加成6.0%
  王语嫣（第4层）- 落霞峰 [异地] - 无法提供加成
```

```java
private String formatBreakthroughResult(BreakthroughResult result) {
    StringBuilder sb = new StringBuilder();
    sb.append(result.getMessage()).append("\n\n");
    sb.append(String.format("突破成功率：%.1f%%\n", result.getSuccessRate()));
    sb.append(String.format("当前境界：第%d层\n", result.getNewLevel()));
    sb.append(String.format("可用属性点：%d\n", result.getFreeStatPoints()));
    sb.append(String.format("下次突破成功率：%.1f%%", result.getNextBreakthroughRate()));
    return sb.toString();
}
```

## 七、架构设计亮点

### 7.1 DDD 分层架构

```
domain/user/
├── entity/DaoProtection.java          # 护道关系实体
├── repository/DaoProtectionRepository.java  # Repository接口(Port)
└── vo/
    ├── DaoProtectionResult.java       # 护道结果VO
    ├── DaoProtectionQueryResult.java  # 查询结果VO
    └── ProtectionInfo.java            # 护道详情VO

infrastructure/
├── mapper/DaoProtectionMapper.java    # MyBatis-Flex Mapper
└── repository/DaoProtectionRepositoryImpl.java  # Repository实现(Adapter)

service/
└── CultivationService.java            # 业务逻辑编排

handle/command/
└── CultivationCommandHandler.java     # 命令处理+响应格式化
```

### 7.2 依赖倒置

- `domain` 层定义 Repository 接口（Port）
- `infrastructure` 层实现 Repository（Adapter）
- `service` 层依赖 `domain` 接口，不依赖具体实现

### 7.3 跨平台设计

- 命令处理器完全平台无关
- 响应格式化为纯文本，适配各平台消息格式
- 平台监听器仅负责IO解析，调用 CommandHandler
