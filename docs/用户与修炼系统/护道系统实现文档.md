# 护道系统实现文档

## 概述

护道系统是 XianTao 的核心社交互助玩法，完全基于道友间的人情羁绊，无功德奖励，无系统收益。本文档详细记录护道系统的技术实现。

---

## 一、核心设计规则

### 1.1 基础约束

| 规则项         | 实现方式              |
|-------------|-------------------|
| **境界压制**    | 护道者境界 `>=` 突破者境界  |
| **护道者数量限制** | 最多同时为 **3名** 道友护道 |
| **突破者数量限制** | 接受无限多护道者          |
| **位置检测**    | 突破瞬间实时校验同地图节点     |

### 1.2 加成计算公式

```text
// 单人加成公式
单人加成 = 5% + (护道者境界层级 -突破者境界层级) × 1%

// 总加成公式
总加成 = min(20%,所有【同地点】护道者加成之和)
```

## 二、数据库设计

### 2.1 表结构

**表名**: `xt_dao_protection`

| 字段             | 类型        | 说明     | 约束                                  |
|----------------|-----------|--------|-------------------------------------|
| `id`           | BIGSERIAL | 护道关系ID | PRIMARY KEY                         |
| `protector_id` | BIGINT    | 护道者ID  | NOT NULL, FK -> xt_user(id)         |
| `protege_id`   | BIGINT    | 被护道者ID | NOT NULL, FK -> xt_user(id)         |
| `create_time`  | TIMESTAMP | 建立时间   | NOT NULL, DEFAULT CURRENT_TIMESTAMP |
| `update_time`  | TIMESTAMP | 更新时间   | NOT NULL, DEFAULT CURRENT_TIMESTAMP |

**唯一约束**: `UNIQUE (protector_id, protege_id)` - 防止重复护道关系

**索引**:

- `idx_dao_protection_protector` - 护道者查询优化
- `idx_dao_protection_protege` - 被护道者查询优化

## 四、核心业务逻辑

### 4.1 建立护道关系

**方法**: `CultivationService.establishProtection()`

**校验流程**:

```
1. 护道者存在性检查
   ↓
2. 被护道者存在性检查（按道号查找）
   ↓
3. 境界压制校验：protector.level >= protege.level
   ↓
4. 护道者名额检查：countByProtectorId < 3
   ↓
5. 重复关系检查：findByProtectorAndProtege
   ↓
6. 创建护道关系并保存
```

### 4.2 突破时护道加成计算

**方法**: `CultivationService.attemptBreakthrough()`

**执行流程**:

```
1. 修为充足性检查
   ↓
2. 计算护道加成（实时位置校验）
   - 查询所有护道者：findByProtegeId(userId)
   - 遍历检查是否同地点：isInSameLocation()
   - 累加同地点护道者的单人加成
   - 截断至20%上限
   ↓
3. 最终成功率 = min(100%, 基础成功率 + 护道加成)
   ↓
4. 随机判定突破成功/失败
   ↓
5. 清除所有护道关系（无论成功失败）
   ↓
6. 保存用户新状态
```

### 4.3 突破后自动解除护道

**方法**: `CultivationService.clearProtegeRelations()`

**设计决策**:

- ✅ 突破成功后自动解除（无论成功/失败）
- ✅ 不通知护道者

### 4.4 解除护道关系

**方法**: `CultivationService.removeProtection()`

**执行流程**:

```
1. 查找被护道者（按道号）
   ↓
2. 查找护道关系
   ↓
3. 删除护道关系
   ↓
4. 返回操作结果
```

### 4.5 护道查询

**方法**: `CultivationService.queryProtectionInfo()`

**查询内容**:

1. **作为护道者**: 显示正在为谁护道
    - 被护道者道号、境界、所在地图
    - 是否同地点标记
    - 单人加成百分比

2. **作为被护道者**: 显示谁在为自己护道
    - 护道者道号、境界、所在地图
    - 是否同地点标记
    - 实际提供加成（异地为0%）
    - 总加成（ capped at 20%）

## 五、命令处理器

### 5.1 专属指令集

| 指令          | 方法                            | 说明          |
|-------------|-------------------------------|-------------|
| `护道 [道号]`   | `handleEstablishProtection()` | 建立护道关系      |
| `护道解除 [道号]` | `handleRemoveProtection()`    | 解除护道关系      |
| `护道查询`      | `handleQueryProtection()`     | 查询护道信息      |
| `突破`        | `handleBreakthrough()`        | 触发突破（含护道加成） |


## 六、边界情况处理

### 6.1 场景测试矩阵

| 场景              | 系统行为                  |
|-----------------|-----------------------|
| 护道者移动到其他地图      | 不解除关系，突破时位置校验失败提供0%加成 |
| 突破者移动到其他地图      | 同上，异地护道者无法提供加成        |
| 护道者已满3人想加第4人    | 拦截并提示先解除部分关系          |
| 突破者接受5人护道，总计30% | 允许关系存在，加成截断至20%       |
| 突破结束（成功/失败）     | 自动解除该突破者的所有护道关系       |
| 护道者/被护道者不存在     | 返回错误提示                |
| 重复建立护道关系        | 拦截并提示已存在关系            |

### 6.2 并发安全

- ✅ 所有护道操作在 `@Transactional` 事务中执行
- ✅ 数据库唯一约束防止重复关系
