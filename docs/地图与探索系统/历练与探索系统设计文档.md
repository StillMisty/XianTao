# 历练与探索系统设计文档

## 1. 模块概述

历练与探索系统是仙道游戏的核心玩法，提供离线收益（AFK）和主动探索两种资源获取方式。

### 1.1 核心职责
- 历练系统：离线挂机收益（懒加载结算）
- 探索系统：主动探索当前区域（即时反馈）
- 体力消耗管理
- 收益计算与分配

### 1.2 技术栈
- **ORM**：MyBatis-Flex
- **设计模式**：懒加载（Lazy Evaluation）
- **AI集成**：Spring AI（探索描述生成）

## 2. 历练系统（AFK收益）

### 2.1 核心设计理念

#### 2.1.1 懒加载结算原理
系统**不实时累积**历练收益，而是在玩家结束历练时一次性计算。

**传统方案 vs 懒加载方案**：

| 对比项 | 传统定时任务 | 懒加载方案 |
|--------|------------|-----------|
| 数据库写入频率 | 每分钟更新所有在线玩家 | 仅在结束时更新一次 |
| 离线支持 | 需额外处理离线逻辑 | 天然支持离线计算 |
| 并发压力 | 高（大量定时任务） | 低（按需计算） |
| 数据一致性 | 需处理竞态条件 | 简单（单次事务） |

#### 2.1.2 历练状态追踪
在xt_user表中记录：
- `training_start_time`：历练开始时间戳
- `location_id`：当前所在地图ID（决定掉落池）

**计算时长**：
```
历练分钟数 = (当前时间 - training_start_time) 的分钟差
```

### 2.2 收益计算公式

#### 2.2.1 铜币收益
```
铜币 = 10 × 历练分钟数 × 效率倍率
效率倍率 = 1.0 + 敏捷 × 0.01
```

**示例**：
- 历练60分钟，敏捷20点
- 效率倍率 = 1.0 + 20×0.01 = 1.2
- 铜币收益 = 10 × 60 × 1.2 = 720铜币

#### 2.2.2 灵石收益
```
灵石 = 5 × (历练分钟数 / 60)  // 按小时计算，不足1小时不计
```

**示例**：
- 历练60分钟 → 5灵石
- 历练119分钟 → 5灵石（不足2小时）
- 历练120分钟 → 10灵石

#### 2.2.3 物品掉落
```
掉落次数 = (历练分钟数 / 10) × 效率倍率
每次掉落从当前地图特产池中随机抽取
```

**示例**：
- 历练60分钟，敏捷20点，效率倍率1.2
- 掉落次数 = (60/10) × 1.2 = 7.2 → 7次
- 每次从地图specialties中加权随机抽取

**物品数量**：
- 每次掉落1-3个（随机）
- 同名物品自动堆叠

#### 2.2.4 经验收益
```
经验 = 10 + (智慧 × 2) + (地图等级 × 5)
```

**示例**：
- 智慧15点，地图等级10
- 经验 = 10 + 15×2 + 10×5 = 90经验

**注意**：经验受存储上限限制（详见用户系统文档）

### 2.3 体力消耗

#### 2.3.1 消耗规则
```
历练体力消耗 = 历练分钟数 × 1点/分钟
```

**示例**：
- 历练60分钟 → 消耗60点体力
- 历练120分钟 → 消耗120点体力

#### 2.3.2 前置检查
开始历练前检查体力是否足够：
- 若不足，返回错误提示
- 不扣减体力，操作不执行
- 引导玩家打坐恢复或使用丹药

### 2.4 历练流程

#### 2.4.1 开始历练
```
接收开始请求
    ↓
验证用户状态（必须为IDLE）
    ↓
检查体力是否足够
    ↓
扣除体力
    ↓
设置training_start_time = 当前时间
    ↓
设置用户状态为TRAINING
    ↓
保存用户数据
    ↓
返回成功消息
```

#### 2.4.2 结束历练
```
接收结束请求
    ↓
检查是否在历练中（training_start_time != null）
    ↓
计算历练分钟数
    ↓
计算各项收益（铜币、灵石、物品、经验）
    ↓
应用收益到用户实体
    ↓
重置training_start_time = 当前时间（新一轮历练开始）
    ↓
设置用户状态为IDLE
    ↓
保存用户数据
    ↓
返回收益详情
```

#### 2.4.3 查询历练状态
```
接收查询请求
    ↓
检查是否在历练中
    ↓
若在 → 计算预估收益（不应用）
    ↓
返回当前历练时长和预估收益
```

## 3. 探索系统

### 3.1 核心设计理念

探索是**主动操作**，玩家消耗固定体力探索当前区域，获得即时反馈和LLM美化描述。

**与历练的区别**：
- 历练：长时间挂机，被动收益
- 探索：单次操作，即时反馈，AI生成描述

### 3.2 有效智慧计算

探索难度受地图等级影响，高等级地图需要更高智慧才能发现稀有物品。

```
有效智慧 = 智慧 - (地图等级 - 1) × 5
最低有效智慧 = 1
```

**示例**：
- 玩家智慧20点，地图等级1
- 有效智慧 = 20 - 0×5 = 20

- 玩家智慧20点，地图等级3
- 有效智慧 = 20 - 2×5 = 10

**设计意图**：
- 鼓励玩家提升智慧属性
- 高等级地图对低智慧玩家不友好
- 平衡不同地图的探索价值

### 3.3 探索结果分级

根据有效智慧划分四个等级：

| 有效智慧范围 | 结果类型 | 奖励内容 | 触发概率 |
|------------|---------|---------|---------|
| 1-20 | 基础发现 | 1个基础材料 | 常见 |
| 21-50 | 稀有发现 | 1个稀有材料 | 中等 |
| 51-80 | 重大发现 | 2个稀有材料 + 30%概率获得配方 | 稀有 |
| 81+ | 惊人发现 | 3个稀有材料 + 必定获得高级配方 + 20%概率获得灵蛋 | 罕见 |

#### 3.3.1 基础发现（有效智慧1-20）
**奖励**：
- 1个基础材料（从地图specialties中随机抽取）
- 数量：1-3个

**示例**：
- 在枯骨林探索，获得"毒龙草 x2"

#### 3.3.2 稀有发现（有效智慧21-50）
**奖励**：
- 1个稀有材料（从预定义稀有列表中随机）
- 数量：1个

**稀有材料列表**：
- 灵草、精铁、灵芝等

#### 3.3.3 重大发现（有效智慧51-80）
**奖励**：
- 2个稀有材料
- 30%概率获得配方（如"初级炼药配方"）

#### 3.3.4 惊人发现（有效智慧81+）
**奖励**：
- 3个稀有材料
- 必定获得高级配方（如"高级炼药配方"）
- 20%概率获得灵蛋（如"火灵蛋"）

### 3.4 体力消耗

```
探索体力消耗 = 20点（固定）
```

**设计意图**：
- 探索是高频操作，固定消耗便于玩家规划
- 相比历练（1点/分钟），探索单位时间消耗更高
- 鼓励玩家根据体力情况选择历练或探索

### 3.5 LLM美化描述

#### 3.5.1 调用时机
探索完成后，调用Spring AI生成个性化描述。

#### 3.5.2 传入参数
```java
ExplorationDescriptionFunction.Request request = new Request(
    mapNode.getName(),      // 地图名称
    result.getEventType(),  // 事件类型
    foundItemNames          // 发现的物品名称列表
);
```

#### 3.5.3 生成示例

**输入**：
- 地图名称：枯骨林
- 事件类型：发现基础材料
- 物品列表：["毒龙草"]

**LLM输出**：
```
您在枯骨林的阴影中仔细搜寻，在一具古老骸骨旁发现了几株散发着幽光的毒龙草。这些灵草似乎吸收了此地的阴气，叶片呈现出诡异的紫色。
```

#### 3.5.4 降级方案
若LLM服务不可用，使用固定模板：
```
您在{地图名称}中探索，发现了{物品名称}。
```

### 3.6 探索流程

```
接收探索请求
    ↓
验证用户状态（必须为IDLE）
    ↓
检查体力是否足够（≥20点）
    ↓
扣除20点体力
    ↓
获取当前地图
    ↓
计算有效智慧
    ↓
根据有效智慧确定结果等级
    ↓
生成奖励（物品、配方、灵蛋）
    ↓
计算经验收益
    ↓
调用LLM生成美化描述
    ↓
应用奖励到用户实体
    ↓
保存用户数据
    ↓
返回探索结果（含LLM描述）
```

## 4. 服务层设计

### 4.1 TrainingService（历练服务）

**职责**：历练收益计算与应用

**核心方法**：

#### 4.1.1 历练管理
- `startTraining(userId)`：开始历练
- `startTrainingWithStaminaCheck(userId, durationMinutes)`：开始历练（带体力检查）
- `endTraining(userId)`：结束历练并应用奖励
- `calculateTrainingRewards(userId)`：计算历练奖励（懒加载）

#### 4.1.2 收益计算
- `calculateEfficiencyMultiplier(agility)`：计算效率倍率
- `calculateCoinsReward(minutesTraining, efficiencyMultiplier)`：计算铜币收益
- `calculateSpiritStonesReward(minutesTraining)`：计算灵石收益
- `calculateItemsReward(minutesTraining, efficiencyMultiplier, mapNode)`：计算物品掉落

#### 4.1.3 奖励应用
- `applyTrainingRewards(userId, rewards)`：应用历练奖励到用户

### 4.2 ExplorationService（探索服务）

**职责**：探索逻辑与LLM集成

**核心方法**：

#### 4.2.1 探索执行
- `exploreCurrentArea(userId)`：探索当前区域（带体力检查）

#### 4.2.2 结果生成
- `generateExplorationResult(user, mapNode)`：生成探索结果
- `calculateExpGained(wisdom, mapLevel)`：计算经验收益

#### 4.2.3 物品查找
- `findBasicItems(mapNode, count)`：查找基础物品
- `findRareItems(mapNode, count)`：查找稀有物品

### 4.3 StaminaService（体力服务）

**职责**：体力消耗与恢复

**核心方法**：
- `checkAndConsumeTrainingStamina(userId, durationMinutes)`：检查并消耗历练体力
- `checkAndConsumeExplorationStamina(userId)`：检查并消耗探索体力
- `getStaminaInfo(userId)`：获取体力信息（含离线恢复计算）

## 5. 性能优化

### 5.1 懒加载优势

#### 5.1.1 减少数据库写入
- 传统方案：每分钟更新所有在线玩家的历练收益
- 懒加载方案：仅在结束时更新一次
- **写入减少99%**

#### 5.1.2 支持真正离线
- 玩家离线后，历练时间继续累积
- 上线后一次性结算，无需补偿逻辑

### 5.2 缓存策略

#### 5.2.1 地图配置缓存
- 地图specialties变化频率低
- 使用Spring Cache缓存地图节点
- 避免重复查询数据库

#### 5.2.2 LLM响应缓存
- 相同地图+相同物品的探索描述可缓存
- 使用LRU缓存，容量1000条
- 缓存Key：`mapName_eventType_itemNames.hashCode()`

## 6. 安全性与校验

### 6.1 输入校验

#### 6.1.1 历练时长校验
- 时长必须 > 0
- 时长不能超过合理范围（如最大1440分钟=24小时）

#### 6.1.2 状态校验
- 用户状态必须为IDLE才能开始历练/探索
- 历练中无法执行其他操作

### 6.2 资源校验

#### 6.2.1 体力检查
- 开始前检查体力是否足够
- 不足时拒绝操作，不扣减体力

#### 6.2.2 经验上限
- 添加经验时检查存储上限
- 超出部分丢弃，提示"修为已达瓶颈"

## 7. 扩展性设计

### 7.1 新收益类型

**步骤**：
1. 在TrainingRewardVO中添加新字段（如"贡献点"）
2. 在calculateTrainingRewards()中计算新收益
3. 在applyTrainingRewards()中应用到用户
4. 更新响应格式化逻辑

### 7.2 新探索结果类型

**步骤**：
1. 调整有效智慧分档
2. 在generateExplorationResult()中添加新分支
3. 添加新的物品/配方/灵蛋类型
4. 更新LLM Prompt模板

### 7.3 历练 buff/debuff

**设计思路**：
- 添加临时增益效果（如"双倍收益符"）
- 在calculateTrainingRewards()中应用倍率
- 支持多种buff叠加
