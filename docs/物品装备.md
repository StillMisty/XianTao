## 2. 装备模板 (xt_equipment_template)

### 2.1 表结构

| `slot` | VARCHAR(32) | 装备部位 (WEAPON/ARMOR/ACCESSORY) |
| `weapon_type` | VARCHAR(32) | 法器子类型，可空（护甲/饰品为 null） |
| `category` | VARCHAR(16) | 法器大类：刀兵/长兵/远兵/奇兵（护甲/饰品为 null） |
| `equip_level` | INT | 装备等级 |
| `base_attack` | INT | 基础攻击 |
| `base_defense` | INT | 基础防御 |
| `base_str` | INT | 基础力量加成 |
| `base_con` | INT | 基础体质加成 |
| `base_agi` | INT | 基础敏捷加成 |
| `base_wis` | INT | 基础智慧加成 |
| `attack_speed` | DECIMAL(3,1) | 攻速（法器专属，影响法决CD恢复速度） |
| `attack_range` | VARCHAR(16) | 近战/远程（法器专属） |
| `drop_weight` | JSONB | 稀有度掉落权重 |
| `create_time` / `update_time` | TIMESTAMP | 时间戳 |

### 2.2 drop_weight JSONB

```json
{"BROKEN": 60, "COMMON": 25, "RARE": 10, "EPIC": 4, "LEGENDARY": 1}
```
稀有度在装备实例化时加权随机决定，key 为 `Rarity.code`（`EnumValue`，与枚举名一致），value 为权重（整数）。通过 `Rarity.roll(dropWeight)` 解析。

### 2.3 装备部位 (EquipmentSlot)

| 枚举值 | 中文名 | 说明 |
|------|------|------|
| WEAPON | 法器 | 主要攻击源，有 weapon_type + category |
| ARMOR | 护甲 | 主要防御源，无 weapon_type |
| ACCESSORY | 饰品 | 属性辅助，无 weapon_type |

```

```

## 4. 装备实例化 —— 稀有度加权随机

### 4.1 流程

装备获取时统一走 `EquipmentService.createEquipment(userId, templateId)`：

```
① 获取 EquipmentTemplate
② 加权随机 roll 出 Rarity（BROKEN/COMMON/RARE/EPIC/LEGENDARY）
③ 在 Rarity 的 qualityMultiplier 范围内随机取一个值
④ 在 Rarity 的 affixCount 范围内随机词条数，从词条池中随机选取（LEGENDARY 额外包含特殊词条）
⑤ 从 Rarity 的 prefixes 列表中随机选前缀 + 模板名 = 装备名
⑥ Equipment.create(...) 并持久化
```

### 4.2 稀有度体系 (Rarity)

| 稀有度 | 品质系数 | 词条数 | 前缀示例 |
|------|---------|--------|---------|
| BROKEN | 0.80–0.99 | 0 | 破烂的、生锈的、残破的、陈旧的 |
| COMMON | 1.00–1.15 | 1 | 普通的、结实的、标准的、常规的 |
| RARE | 1.16–1.35 | 1–2 | 锋利的、精工的、锐利的、精制的 |
| EPIC | 1.36–1.60 | 2–3 | 卓越的、附魔的、辉煌的、不凡的 |
| LEGENDARY | 1.61–2.00 | 3–4 | 传说的、完美的、神话的、神级的 |

### 4.3 属性计算公式

```
攻击 = 基础攻击 + floor(基础攻击 × qualityMultiplier) + forgeLevel × 5
防御 = 基础防御 + floor(基础防御 × qualityMultiplier) + forgeLevel × 5
总属性 = 基础属性 + 属性词条加成
```

## 8. 背包管理

- 装备：每件独立实例，不堆叠
- 材料/消耗品：按模板合并，受 `max_stack` 限制
- 添加物品时自动按 `template_id` 合并
- 历练结算掉落物**自动进背包**，无需手动拾取

## 9. 物品使用系统

### 9.1 统一使用指令

```
使用 [物品名] [参数]
```

支持自动识别物品类型并执行对应效果：

| 物品类型 | 示例 | 效果 |
|---------|------|------|
| 丹药 (POTION) | `使用 天元丹` | 服用丹药，获得效果 |
| 法决玉简 (SKILL_JADE) | `使用 御剑术玉简` | 学习法决 |
| 丹方卷轴 (RECIPE_SCROLL) | `使用 天元丹方` | 学习丹方 |
| 进化石 (EVOLUTION_STONE) | `使用 进化石 1` | 灵兽进化（需指定位置） |

### 9.2 策略模式架构

```
domain/item/handler/
├── ItemUseHandler.java           # 接口
├── PillUseHandler.java           # 丹药处理器 → PillConsumptionService.takePill()
├── SkillJadeUseHandler.java      # 法决玉简处理器 → SkillService.learnFromJade()
├── RecipeScrollUseHandler.java   # 丹方卷轴处理器 → PillRecipeService.learnRecipe()
└── EvolutionStoneUseHandler.java # 进化石处理器 → BeastService.evolveBeast()
```

### 9.3 ItemUseHandler 接口

接口定义：
- `supports(ItemType, ItemTemplate)` → boolean：判断处理器是否支持该物品类型
- `use(Long userId, StackableItem, ItemTemplate, String args)` → String：执行使用逻辑
- `consumesInternally()` → boolean（默认 false）：是否自行管理物品消耗（false 则由 `ItemUseService` 统一扣减）

- `PillUseHandler`、`RecipeScrollUseHandler`：`consumesInternally()` 返回 `false`，由 `ItemUseService` 自动扣减数量
- `SkillJadeUseHandler`、`EvolutionStoneUseHandler`：`consumesInternally()` 返回 `true`，自行管理消耗

### 9.4 扩展方式

创建新的 Handler 实现 `ItemUseHandler` 接口并添加 `@Component` 注解即可自动注册：

创建新的 `@Component` 类实现 `ItemUseHandler` 接口即可自动注册。需实现：
- `supports()`：返回 `true` 当物品类型匹配
- `use()`：执行使用逻辑
