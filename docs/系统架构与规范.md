# 系统架构与规范

## 1. 整体架构

严格遵循 **领域驱动设计 (DDD) + 端口适配器 (Ports & Adapters)** 模式。

```
┌─────────────────────────────────────────────┐
│                  Handle 层                    │
│  (onebotv11/)  (web/)  (command/)             │ ← 纯 I/O 解析 + 文本渲染
├─────────────────────────────────────────────┤
│                  Service 层                    │
│  认证 + 编排 + 业务逻辑 (返回 ServiceResult<T>)  │
├─────────────────────────────────────────────┤
│                  Domain 层                    │
│  entity/  enums/  repository/  vo/           │ ← 纯领域模型，无基础设施依赖
├─────────────────────────────────────────────┤
│              Infrastructure 层                 │
│  mapper/  repository/  mybatis/              │ ← 适配器实现
└─────────────────────────────────────────────┘
```

### 1.1 依赖规则

- `domain` → 纯 Java，不依赖 infrastructure 或 handle
- `service` → 依赖 domain，调用 infrastructure 接口
- `handle` → 只调用 service，不包含业务逻辑
- `infrastructure` → 实现 domain 定义的接口

### 1.2 命令处理流程

```
平台监听器 (OneBotV11/Web)
  ↓ 解析平台输入
Service 层 (认证 + 业务逻辑)
  ↓ 返回 ServiceResult<T>
Command Handler (格式化 VO → 文本)
  ↓ 纯文本
平台监听器 (渲染视图)
```

未来可插入 LLM 层做战斗叙事美化（格式化为修仙小说风格文本）。

## 2. 核心架构规则

### 2.1 ServiceResult<T> — 统一返回类型

所有 Service 层公开方法返回 `ServiceResult<T>`（sealed 接口）：

- `Success<T>(T data)` — 认证通过 + 业务执行，承载领域 VO
- `Failure<T>(String errorCode, String errorMessage)` — 认证失败或业务校验不通过，承载错误码和错误消息
- 工厂方法：`authFailure()` 返回 `AUTH_FAILED` 错误，`businessFailure()` 返回 `BUSINESS_ERROR` 错误

Handler 层通过 pattern matching 分支处理，禁止 null 检查。

### 2.2 Service 双层 API

每个 `*Service.java` 公开两种方法：

- **Public**: `(PlatformType, String openId, ...)` → 认证后委托 internal → `ServiceResult<T>`
- **Internal**: `(Long userId, ...)` → 直接执行业务 → raw VO

Internal 方法是 package-private 或 public（便于测试）。

### 2.3 懒加载状态解析 (UserStateService)

所有服务通过 `userStateService.loadUser(userId)` 统一加载用户实体，内部自动解析过期的运行时状态：

```
UserStateService.loadUser(userId)
  └─ resolveState(user)
       ├─ tryCompleteTravel(user)    // 旅行到期 → IDLE + 更新位置
       ├─ tryDyingRecovery(user)     // 濒死超时 → 恢复 20% HP + IDLE
       ├─ tryHpRecovery(user)        // IDLE 时每5分钟恢复 1% HP
       └─ tryExpireBuffs(user)       // 删除过期 pill buff（dirty=false，不触发 save）
```

方法说明：
- `loadUser(userId)` — 加载用户并自动解析过期状态（旅行结算、濒死恢复、HP恢复、buff清理）
- `save(user)` — 保存用户，统一入口
- `resolveState(user)` — 内部方法，按顺序执行过期状态解析，仅在发生变更时自动保存

**设计原则**：
- **脏标记**：`resolveState()` 返回 boolean，仅在确实修改了 user 时才 save
- **早退**：每个 `try*` 方法顶部做条件判断，大部分调用零副作用
- **可扩展**：新增自动结算逻辑只需追加一个 `try*` 方法
- **替代散落调用**：所有 `userRepository.findById(userId).orElseThrow()` → `userStateService.loadUser(userId)`

### 2.4 AuthenticationService

统一身份认证，三阶方法：

| 方法 | 返回 | 校验 |
|------|------|------|
| `authenticate` | `userId` | 仅验证平台+openId 存在 |
| `authenticateAndValidateUser` | `User` | 验证用户存在且状态正常 |
| `authenticateAndValidateStatus` | `User` | 验证用户存在且为指定状态 |

### 2.5 AuthenticatedAspect

`@Around` 拦截标注 `@Authenticated` 的方法。约定前两个参数为 `(PlatformType, String openId)`，可选的第三个参数为 `UserStatus`（要求用户处于指定状态）。调用 `authService.authenticate(platform, openId, requiredStatus)` 完成认证，将 userId 绑定到 `ScopedValue`（`UserContext.CURRENT_USER`）。认证失败返回 `ServiceResult.Failure("AUTH_FAILED", message)`；方法内部抛 `IllegalStateException`/`IllegalArgumentException` 返回原始消息，其余异常返回 `"系统繁忙，请稍后再试"`。

### 2.6 UserContext

使用 `ScopedValue` 存储当前认证用户 ID。Internal 方法通过 `getCurrentUserId()` 获取当前用户 ID，无需从参数传递。`ScopedValue` 替代传统 `ThreadLocal`，支持结构化并发。

## 3. Java 25 特性规范

| 特性 | 使用场景 | 要求 |
|------|---------|------|
| Virtual Threads | 所有 handle/service 方法 | 兼容，避免 ThreadLocal 阻塞 |
| Records | VO/DTO | 所有值对象优先使用 |
| Sealed Classes | ServiceResult, 枚举扩展 | 编译器 exhaustiveness 检查 |
| Pattern Matching | ServiceResult switch 分支 | 禁止 null 判断 |
| ScopedValue | 用户上下文传递 | 替换 ThreadLocal，支持结构化并发 |

---

## 4. 数据库规范

- **JSONB 优先**: 扩展性结构（掉落池、词条、标签、地图配置等）使用 JSONB 列
- **GIN 索引**: JSONB 字段查询路径创建 GIN 索引
- **MyBatis-Flex**: 使用 `JsonbTypeHandler` 映射 JSONB
- **Flyway**: 所有 schema 变更通过迁移脚本管理

## 6. Item Use 策略模式

- 接口：`ItemUseHandler`（`domain/item/handler/`）
- 方法：`supports(ItemType, ItemTemplate)` / `use(Long userId, StackableItem, ItemTemplate, String args)`
- `consumesInternally()`：默认返回 `false`（由 `ItemUseService` 统一扣减数量），返回 `true` 的处理器自行管理消耗
- 现有实现（4个）：`PillUseHandler`、`SkillJadeUseHandler`、`RecipeScrollUseHandler`、`EvolutionStoneUseHandler`

## 7. 开发工作流

1. **Domain**: 创建 Entity、VO、Repository 接口（Entity 必须含行为方法，VO 用 `record`）
2. **Database**: 编写 Flyway 迁移脚本（`V1.0.XX__description.sql`，含 DML + `COMMENT ON`）
3. **Infra**: 实现 Mapper 和 RepositoryImpl（使用 `mapper.insertOrUpdateSelective()`）
4. **Logic**: Service 层 internal 方法（纯业务，`UserContext.getCurrentUserId()` 取用户 ID）
5. **Auth**: Service 层 public 入口方法（标注 `@Authenticated`）
6. **Command**: handle/command/XXXCommandHandler（VO → 文本格式化）
7. **Entry**: 平台监听器触发
