# 系统架构与规范

## 1. 整体架构

严格遵循 **领域驱动设计 (DDD) + 端口适配器 (Ports & Adapters)** 模式。

```
┌─────────────────────────────────────────────┐
│                  Handle 层                    │
│  (onebotv11/)  (qq/)  (web/)  (command/)     │ ← 纯 I/O 解析 + 文本渲染
├─────────────────────────────────────────────┤
│                  Service 层                    │
│  认证 + 编排 + 业务逻辑 (返回 ServiceResult<T>)  │
├─────────────────────────────────────────────┤
│                  Domain 层                    │
│  entity/  enums/  repository/  vo/           │ ← 纯领域模型，无基础设施依赖
├─────────────────────────────────────────────┤
│              Infrastructure 层                 │
│  mapper/  repository/  mybatis/              │ ← 适配器实现
└─────────────────────────────────────────────┘
```

### 1.1 依赖规则

- `domain` → 纯 Java，不依赖 infrastructure 或 handle
- `service` → 依赖 domain，调用 infrastructure 接口
- `handle` → 只调用 service，不包含业务逻辑
- `infrastructure` → 实现 domain 定义的接口

### 1.2 命令处理流程

```
平台监听器 (OneBotV11/QQ/Web)
  ↓ 解析平台输入
Service 层 (认证 + 业务逻辑)
  ↓ 返回 ServiceResult<T>
Command Handler (格式化 VO → 文本)
  ↓ 纯文本
平台监听器 (渲染视图)
```

未来可插入 LLM 层做战斗叙事美化（格式化为修仙小说风格文本）。

## 2. 核心架构规则

### 2.1 ServiceResult<T> — 统一返回类型

所有 Service 层公开方法返回 `ServiceResult<T>`（sealed 接口）：

```java
sealed interface ServiceResult<T> permits Success, Failure {}

record Success<T>(T data) implements ServiceResult<T> {}
record Failure<T>(String errorMessage) implements ServiceResult<T> {}
```

Handler 层通过 pattern matching 分支处理，禁止 null 检查。

### 2.2 Service 双层 API

每个 `*Service.java` 公开两种方法：

- **Public**: `(PlatformType, String openId, ...)` → 认证后委托 internal → `ServiceResult<T>`
- **Internal**: `(Long userId, ...)` → 直接执行业务 → raw VO

### 2.3 AuthenticationService

统一身份认证，三阶方法：

| 方法 | 返回 | 校验 |
|------|------|------|
| `authenticate` | `userId` | 仅验证平台+openId 存在 |
| `authenticateAndValidateUser` | `User` | 验证用户存在且状态正常 |
| `authenticateAndValidateStatus` | `User` | 验证用户存在且为指定状态 |

## 3. Java 25 特性规范

| 特性 | 使用场景 | 要求 |
|------|---------|------|
| Virtual Threads | 所有 handle/service 方法 | 兼容，避免 ThreadLocal 阻塞 |
| Records | VO/DTO | 所有值对象优先使用 |
| Sealed Classes | ServiceResult, 枚举扩展 | 编译器 exhaustiveness 检查 |
| Pattern Matching | ServiceResult switch 分支 | 禁止 null 判断 |

---

## 4. 数据库规范

- **JSONB 优先**: 扩展性结构（掉落池、词条、标签、地图配置等）使用 JSONB 列
- **GIN 索引**: JSONB 字段查询路径创建 GIN 索引
- **MyBatis-Flex**: 使用 `PgJsonbTypeHandler` 映射 JSONB
- **Flyway**: 所有 schema 变更通过迁移脚本管理

## 6. 开发工作流

1. **Domain**: 创建 Entity、VO、Repository 接口
2. **Database**: 编写 Flyway 迁移脚本
3. **Infra**: 实现 Mapper 和 RepositoryImpl
4. **Logic**: Service 层 internal 方法（纯业务）
5. **Auth**: Service 层 public 入口方法
6. **Command**: handle/command/XXXCommandHandler
7. **Entry**: 平台监听器触发

## 7. 三大新系统概览

### 7.1 装备系统改造

- `xt_equipment_template` → 装备专用模板表，武器有 `weapon_type` + `category`
- 稀有度在实例化时加权随机，`ItemTemplate.rarity` 已移除
- "武器" → "法器"
- 详见 [物品装备.md](./物品装备.md)

### 7.2 怪物 + 自动战斗

- `xt_monster` + `xt_map_monster` → 怪物模板 + 地图关联
- `CombatService.simulateTraining()` → 懒结算，一次性模拟所有战斗
- 法器类型克制怪物类型，×1.5 / ×0.8
- HP 归零 → DYING 濒死状态
- 详见 [地图旅行历练.md](./地图旅行历练.md)

### 7.3 法决系统

- `xt_spell` + `xt_user_spell` → 法决模板 + 已习得
- 只有攻击类效果，零消耗，独立 CD
- 法器绑定（精确/大类/不绑定），不绑定效果弱于绑定
- 槽位制：初始 1 槽，每 20 级 +1
- 详见 [法决.md](./法决.md)
