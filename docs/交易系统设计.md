# 交易系统设计

## 1. 设计原则

- **文字挂机 MUD**：所有交互异步、无实时、无定时任务
- **事件驱动**：所有价格/库存变动均由玩家交互触发（懒检查），无定时任务
- **LLM 优先**：商铺掌柜为 LLM NPC，所有交易操作通过自然语言触发
- **真实感**：价格受世界事件影响，库存受供需调节

---

## 2. 物品可交易性

### 2.1 `ItemTemplate.tradable`

```sql
ALTER TABLE xt_item_template ADD COLUMN tradable BOOLEAN NOT NULL DEFAULT TRUE;
```

- 控制**这类物品能否进入市场**
- `StackableItem` 无 per-instance 覆写，完全继承模板
- `Equipment` 有 per-instance tradable，`equip()` 时设 `false` 不可逆转

| 物品 | tradable | 原因 |
|------|----------|------|
| 聚气丹 | true | 消耗品自由流通 |
| 玄铁剑（未装备） | true | 可交易 |
| 玄铁剑（已装备） | false | 穿戴即绑定 |
| 秘境钥匙 | false | 控制进度 |

### 2.2 装备状态

```sql
ALTER TABLE xt_equipment ADD COLUMN status VARCHAR(16) NOT NULL DEFAULT 'OWNED'
    CHECK (status IN ('OWNED', 'EQUIPPED', 'LISTED', 'SOLD'));
```

上架 → `LISTED`，成交 → `SOLD`，取消 → `OWNED`，装备 → `EQUIPPED`。

所有使用/装备操作前检查 `status != 'LISTED'`，杜绝已挂售装备被误用。

## 3. 玩家交易市场

### 3.1 `market_listing` 表

```sql
CREATE TABLE market_listing (
    id            BIGSERIAL PRIMARY KEY,
    seller_id     BIGINT NOT NULL REFERENCES xt_user(id),

    -- STACKABLE / EQUIPMENT
    listing_type  VARCHAR(16) NOT NULL CHECK (listing_type IN ('STACKABLE', 'EQUIPMENT')),
    template_id   BIGINT NOT NULL REFERENCES xt_item_template(id),
    unit_price    BIGINT NOT NULL CHECK (unit_price > 0),
    quantity      INT NOT NULL CHECK (quantity > 0),

    -- STACKABLE → {"propertiesHash": 12345}
    -- EQUIPMENT  → {"equipmentId": 456, "forgeLevel": 3, "rarity": "EPIC"}
    details       JSONB NOT NULL DEFAULT '{}'::jsonb,

    status        VARCHAR(16) NOT NULL DEFAULT 'ACTIVE'
                  CHECK (status IN ('ACTIVE', 'SOLD', 'CANCELLED')),
    created_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

`quantity` 通用字段：STACKABLE 为挂售数量，EQUIPMENT 恒为 1。
`details` JSONB 保持扩展性，未来加新类型（BEAST_EGG、SKILL）不改表。

### 3.2 上架—原子扣库存

**堆叠物品**：
```sql
UPDATE xt_inventory_item
SET quantity = quantity - #{qty}
WHERE user_id = #{sellerId}
  AND template_id = #{templateId}
  AND properties_hash = #{hash}
  AND quantity >= #{qty};
```
上架即从库存扣除，杜绝"挂售后用掉"。

**装备**：
```sql
UPDATE xt_equipment SET status = 'LISTED' WHERE id = #{equipId} AND status = 'OWNED';
```

### 3.3 成交/取消

**成交**：买方向系统支付灵石 → 系统记卖方应收 → 买方获得物品
- 堆叠：`INSERT INTO xt_inventory_item`（新行） 或 `UPDATE quantity`（已有同类行）
- 装备：`UPDATE xt_equipment SET user_id = #{buyerId}, status = 'OWNED'`

**取消**：归还库存
- 堆叠：`UPDATE quantity = quantity + #{qty}`，无匹配行则 INSERT
- 装备：`UPDATE status = 'OWNED'`

### 3.4 交易税

成交时系统扣除一定比例灵石（建议 5-10%），作为灵石回收的主要 sink 之一。

## 4. 商铺系统

### 4.1 `shop_npc` — 掌柜模板

```sql
CREATE TABLE shop_npc (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(64) NOT NULL,        -- 掌柜名
    map_node_id     BIGINT NOT NULL REFERENCES xt_map_node(id),
    personality     VARCHAR(16),                 -- MBTI 等，复用地灵模式
    category_multiplier JSONB,                   -- {"HERB": 0.85, "ORE": 1.15} NPC 个性定价
    system_prompt   TEXT,                        -- LLM system prompt
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

`category_multiplier` 让不同地图的商铺同物不同价：
- 东洲药店：`{"HERB": 0.85, "POTION": 0.9, "ORE": 1.15}` → 药材便宜矿石贵
- 西洲铁匠铺：`{"ORE": 0.8, "EQUIPMENT": 0.9, "HERB": 1.2}` → 矿石便宜药材贵

`system_prompt` 注入世界观 + 经济事件，使 LLM 能自然解释价格波动。

### 4.2 `shop_product` — 商品目录

```sql
CREATE TABLE shop_product (
    id              BIGSERIAL PRIMARY KEY,
    shop_npc_id     BIGINT NOT NULL REFERENCES shop_npc(id),
    template_id     BIGINT NOT NULL REFERENCES xt_item_template(id),
    base_price      BIGINT NOT NULL,             -- 基准价
    min_price       BIGINT NOT NULL,             -- 最低价
    max_price       BIGINT NOT NULL,             -- 最高价
    min_stock       INT NOT NULL DEFAULT 0,      -- 最低库存
    max_stock       INT NOT NULL,                -- 最高库存

    -- 运行时状态（非定时任务维护）
    current_price   BIGINT NOT NULL,
    current_stock   INT NOT NULL,
    last_sale_time  TIMESTAMP,                   -- 最近售出/补货/调价时间
    version         INT NOT NULL DEFAULT 0        -- 乐观锁

    -- 约束
    CONSTRAINT chk_price CHECK (
        min_price <= base_price AND base_price <= max_price AND
        min_price <= current_price AND current_price <= max_price
    ),
    CONSTRAINT chk_stock CHECK (
        min_stock <= current_stock AND current_stock <= max_stock
    )
);

CREATE INDEX idx_shop_product_npc ON shop_product(shop_npc_id);
```

每人每日限购记录存在 `User.extraData` JSONB 中（不另建表）。

### 4.3 懒补货 / 懒调价 — 两条规则

所有检查在玩家**打开商铺或尝试购买时触发**，无定时任务。

#### 规则1：缺货涨价补货

```
触发条件：current_stock = 0
          AND (now - last_sale_time) > restockInterval (配置，如 2 小时)

动作：
  current_stock = MIN(current_stock + restockQty(配置, 如 50), max_stock)
  current_price = MIN(current_price × priceSurgeMultiplier(如 1.15), max_price)
  last_sale_time = now
```

#### 规则2：滞销降价减仓

```
触发条件：current_stock > threshold (如 max_stock × 0.8)
          AND (now - last_sale_time) > decayInterval (配置，如 6 小时)

动作：
  current_stock = MAX(current_stock - decayQty(如 20), min_stock)
  current_price = MAX(current_price × priceDecayMultiplier(如 0.92), min_price)
  last_sale_time = now
```

#### 效果

| 状态 | 系统反应 | 玩家感知 |
|------|---------|---------|
| 热销品卖空 | 几小时后加量涨价 | "怎么贵了？" |
| 冷门品滞销 | 降价减仓，有人买即止跌 | "这玩意儿终于降价了" |
| 人多服抢购 | 频繁触发补货涨价循环，直到 max_price 见顶 | 先到先得的紧迫感 |
| 人少服无人问 | 持续降价到 min_price，少量库存自然消化 | 低价淘货的爽感 |

### 4.4 调货机制

当玩家询问某物品 → LLM 发现店内无此商品 → 可主动提出调货。

**`shop_special_order` 表**：

```sql
CREATE TABLE shop_special_order (
    id              BIGSERIAL PRIMARY KEY,
    player_id       BIGINT NOT NULL REFERENCES xt_user(id),
    shop_npc_id     BIGINT NOT NULL REFERENCES shop_npc(id),
    template_id     BIGINT NOT NULL REFERENCES xt_item_template(id),
    unit_price      BIGINT NOT NULL,             -- base_price × sourcingMarkup(如 1.3)
    quantity        INT NOT NULL DEFAULT 1,
    deposit         BIGINT NOT NULL,             -- 定金 (unitPrice × 10%)
    status          VARCHAR(16) NOT NULL DEFAULT 'PENDING'
                    CHECK (status IN ('PENDING', 'READY', 'COLLECTED', 'CANCELLED')),
    sourcing_hours  INT NOT NULL,                -- 调货所需小时数
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

**流程**：

```
LLM: "客官，此物小店暂无，但可托人从天机阁调货，
      需 X 个时辰，价格比市价贵三成，客官可要？"
        ↓
玩家同意 onCue("调货") / 自然语言同意
        ↓
系统扣 10% 定金，INSERT special_order，status = 'PENDING'
        ↓
玩家下次进店主动提起 → LLM 调用 checkSpecialOrder()
  ├── 时间未到 → "还在路上，约莫还有 X 个时辰"
  └── 时间已到 → status = 'READY'
                  玩家补尾款取货 → status = 'COLLECTED'
                  尾款逾期未付 → 定金不退，status = 'CANCELLED'
```

## 5. 价格引擎

### 5.1 最终价格公式

```
currentPrice = basePrice × worldEventMultiplier × shopkeeperMultiplier

  worldEventMultiplier   = 1.0 + Σ(活跃事件对当前物品的影响)
  shopkeeperMultiplier   = npc.categoryMultiplier[match(item.category)] ?? 1.0
```

### 5.2 世界事件

#### `world_event` 表

```sql
CREATE TABLE world_event (
    id                BIGSERIAL PRIMARY KEY,
    title             VARCHAR(128) NOT NULL,     -- "东洲灵脉动荡"
    description       TEXT NOT NULL,             -- LLM生成的叙事描述
    affected_tags     JSONB NOT NULL,            -- ["HERB", "POTION"] 影响物品标签
    global_multiplier NUMERIC(4,2) NOT NULL,     -- 未命中标签物品的通用波动 (1.02-1.05)
    affected_min      NUMERIC(4,2) NOT NULL,     -- 受影响物品下限 (0.5)
    affected_max      NUMERIC(4,2) NOT NULL,     -- 受影响物品上限 (2.0)
    start_time        TIMESTAMP NOT NULL,
    end_time          TIMESTAMP NOT NULL,
    created_by        VARCHAR(32) DEFAULT 'SYSTEM'
);
```

**事件生成**：cron 触发（这是极少数使用定时任务的地方——世界事件是系统层面的心跳），每天 1-2 次。LLM 生成事件标题、描述和影响范围。

**价格写入**：商铺 `current_price` 的懒检查中，计算 `worldEventMultiplier` 时从 DB 读取当前活跃事件。

**掌柜记忆**：活跃事件注入 `shop_npc.system_prompt`，使 LLM 能给出自然语言解释：

> "客官有所不知，东洲灵脉动荡，这几日药材价格自然水涨船高..."

## 6. 旅行商人（扩展预留）

设计原则：
- 旅行事件触发（复用 `travel_events` 系统）
- 有限库存，稀有物品概率出现
- 价格浮动倍数更大（0.5x - 3.0x）
- 开放 LLM 对话 + 讨价还价（扩展 `haggle()` 工具）
- 不建表，纯事件驱动，未来按需落地

## 7. 开发流程

参照通用模式：

1. `domain/market/` — `MarketListing` entity、`ListingStatus` 枚举
2. `domain/shop/` — `ShopNpc`、`ShopProduct`、`ShopSpecialOrder` entities
3. `domain/item/ItemTemplate` — 加 `tradable` 字段
4. `db/migration/` — 建表 + ALTER + DML 种子数据
5. `infrastructure/mapper/` — Mapper 接口
6. `infrastructure/repository/` — RepositoryImpl
7. `service/` — `MarketService`、`ShopService`、`PriceService`、`WorldEventService`
8. `service/ai/` — `ShopTools`（`@Tool` 函数）、`ShopChatService`（复用 SpiritChatService 模式）
9. `handle/command/` — 命令格式化器
10. `handle/listener/` — 双平台监听器
