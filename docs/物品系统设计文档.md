# 物品系统设计文档

## 1. 模块概述

物品系统是仙道游戏的资源管理核心，负责装备生成、背包管理、物品掉落等关键功能。

### 1.1 核心职责
- 物品模板配置（装备、消耗品、种子）
- 装备实例化与词条生成
- 背包管理（堆叠、容量限制）
- 装备穿戴与属性计算
- 物品掉落与品质判定

### 1.2 技术栈
- **ORM**：MyBatis-Flex
- **数据库**：PostgreSQL 18 (JSONB存储装备词条和标签)
- **算法**：加权随机算法、动态属性计算

## 2. 数据模型设计

### 2.1 物品模板 (xt_item_template)

#### 2.1.1 表结构说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGSERIAL | 模板ID（主键） |
| name | VARCHAR(128) | 物品名称 |
| type | VARCHAR(32) | 物品类型（EQUIPMENT/CONSUMABLE/SEED等） |
| rarity | VARCHAR(32) | 稀有度（默认common） |
| slot | VARCHAR(32) | 装备部位（仅装备类） |
| equip_level | INT | 装备等级（仅装备类，用于计算词条数值） |
| base_stat_bonus | JSONB | 基础属性加成 |
| base_attack | INT | 基础攻击力 |
| base_defense | INT | 基础防御力 |
| drop_weight | JSONB | 掉落权重配置（仅装备类） |
| tags | JSONB | 物品标签（用于AI检索） |
| grow_time | INT | 生长时间（小时，仅种子/灵蛋） |
| yield_id | VARCHAR(64) | 成熟后产出的物品模板ID（仅种子/灵蛋） |
| survive_rate | INT | 存活率百分比（仅种子/灵蛋） |
| max_stack | INT | 最大堆叠数量（默认1） |
| description | TEXT | 物品描述 |
| create_time | TIMESTAMP | 创建时间 |
| update_time | TIMESTAMP | 更新时间 |

#### 2.1.2 JSONB字段结构

**base_stat_bonus（基础属性加成）**：
```json
{"str": 5, "con": 3, "agi": 2, "wis": 0}
```

**drop_weight（掉落权重配置）**：
```json
{
  "BROKEN": 50,
  "COMMON": 30,
  "RARE": 15,
  "EPIC": 4,
  "LEGENDARY": 1
}
```
- 控制不同稀有度的生成概率
- 总和不一定为100，按相对权重计算

**tags（物品标签）**：
```json
["ore", "metal", "forge_base"]
```
- 用于AI检索和NPC交互的语义标签
- 支持GIN索引加速查询

#### 2.1.3 索引策略
- **idx_item_template_tags**：对tags字段创建GIN索引（标签搜索）

### 2.2 装备实例 (xt_equipment)

#### 2.2.1 表结构说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGSERIAL | 装备唯一ID（主键） |
| user_id | BIGINT | 持有者用户ID（外键） |

| name | VARCHAR(128) | 实际名称（可能因前缀/后缀变化） |

| rarity | VARCHAR(32) | 稀有度 |
| stat_bonus | JSONB | 四维属性加成 |
| attack_bonus | INT | 攻击加成 |
| defense_bonus | INT | 防御加成 |
| equipped | BOOLEAN | 是否已装备 |
| quality_multiplier | DOUBLE PRECISION | 品质乘数 |
| affixes | JSONB | 随机词条 |
| forge_level | INT | 锻造等级（默认0） |
| create_time | TIMESTAMP | 创建时间 |

#### 2.2.2 JSONB字段结构

**stat_bonus（属性加成）**：
```json
{"str": 10, "con": 5, "agi": 3, "wis": 0}
```

**affixes（随机词条）**：
```json
{
  "prefix": "锋利的",
  "suffix": "之力",
  "extra_stats": {"str": 5},
  "special_effects": ["critical_rate +5%"]
```

## 3. 核心业务规则

### 3.1 装备稀有度体系

#### 3.1.1 稀有度分级

| 稀有度 | 英文名 | 颜色标识 | 基础倍率 |
|--------|--------|---------|---------|
| 破损 | BROKEN | 灰色 | 0.5x |
| 普通 | COMMON | 白色 | 1.0x |
| 稀有 | RARE | 绿色 | 1.5x |
| 史诗 | EPIC | 蓝色 | 2.5x |
| 传说 | LEGENDARY | 橙色 | 4.0x |

#### 3.1.2 稀有度生成算法

根据物品模板的drop_weight配置进行加权随机：

**示例配置**：
```json
{"BROKEN": 50, "COMMON": 30, "RARE": 15, "EPIC": 4, "LEGENDARY": 1}
```

总权重 = 50 + 30 + 15 + 4 + 1 = 100

**生成概率**：
- 破损：50/100 = 50%
- 普通：30/100 = 30%
- 稀有：15/100 = 15%
- 史诗：4/100 = 4%
- 传说：1/100 = 1%

### 3.2 装备属性计算

#### 3.2.1 基础属性
从物品模板继承：
- base_attack：基础攻击力
- base_defense：基础防御力
- base_stat_bonus：基础四维属性加成

#### 3.2.2 品质修正
```
最终攻击力 = 基础攻击力 × 品质乘数
最终防御力 = 基础防御力 × 品质乘数
最终属性加成 = 基础属性加成 × 品质乘数
```

**品质乘数来源**：
- 由稀有度决定（见3.1.1表格）
- 可通过锻造提升

#### 3.2.3 随机词条生成

**词条池**：
- 前缀词条：如"锋利的"（+攻击力）、"坚固的"（+防御力）
- 后缀词条：如"之力"（+力量）、"之心"（+体质）
- 特殊效果：如"暴击率+5%"、"闪避率+3%"

**生成规则**：
1. 根据装备等级确定词条数量
   - 1-10级：0-1个词条
   - 11-20级：1-2个词条
   - 21-30级：2-3个词条
   - 31+级：3-4个词条

2. 从词条池中随机抽取
3. 应用到装备的affixes字段

### 3.3 装备部位系统

#### 3.3.1 部位定义

| 部位 | 英文标识 | 说明 |
|------|---------|------|
| 头部 | HEAD | 头盔、帽子 |
| 胸部 | CHEST | 铠甲、衣服 |
| 腿部 | LEGS | 裤子、裙甲 |
| 手部 | HANDS | 手套 |
| 足部 | FEET | 鞋子、靴子 |
| 主手 | MAIN_HAND | 武器 |
| 副手 | OFF_HAND | 盾牌、副武器 |

#### 3.3.2 穿戴约束
- 每个部位只能装备一件装备
- 更换装备时自动卸下原装备
- 卸下后返回背包

### 3.4 背包管理系统

#### 3.4.1 背包容量
- 初始容量：50格
- 可通过道具或等级提升扩容

#### 3.4.2 堆叠规则

**可堆叠物品**：
- 消耗品（丹药、材料）
- max_stack > 1的物品模板

**不可堆叠物品**：
- 装备（每件独立实例）
- max_stack = 1的物品

**堆叠上限**：
- 由物品模板的max_stack字段决定
- 示例：灵石max_stack=9999，丹药max_stack=99

#### 3.4.3 背包操作

**添加物品**：
1. 检查是否为可堆叠物品
2. 若可堆叠，查找已有同类型物品
3. 若存在且未满，增加数量
4. 若不存在或已满，占用新格子
5. 若背包已满，返回失败

**移除物品**：
1. 检查物品是否存在
2. 若为堆叠物品，减少数量
3. 若数量为0，移除该格子
4. 若为装备，直接移除

## 4. 服务层设计

### 4.1 ItemService

**职责**：物品管理核心逻辑

**核心方法**：

#### 4.1.1 装备管理
- `equipItem(userId, itemName)`：穿戴装备
- `unequipItem(userId, slotName)`：卸下装备
- `getCharacterStatus(userId)`：获取角色完整状态（含装备加成）

#### 4.1.2 背包管理
- `getInventorySummary(userId)`：获取背包摘要
- `addItem(userId, itemId, quantity)`：添加物品到背包
- `removeItem(userId, itemId, quantity)`：从背包移除物品

#### 4.1.3 装备生成
- `generateEquipment(templateId, rarity)`：生成装备实例
- `calculateAffixes(equipLevel, rarity)`：计算随机词条

### 4.2 属性变化追踪

#### 4.2.1 AttributeChange VO
记录装备穿戴/卸下时的属性变化：

```java
public record AttributeChange(
    int strChange,
    int conChange,
    int agiChange,
    int wisChange,
    int attackChange,
    int defenseChange,
    int maxHpChange
) {}
```

#### 4.2.2 属性计算流程

**穿戴装备时**：
1. 检查部位是否已有装备
2. 若有，先卸下原装备（应用负向属性变化）
3. 穿戴新装备（应用正向属性变化）
4. 更新用户实体的装备加成
5. 重新计算战斗属性（攻击、防御、HP上限）

**卸下装备时**：
1. 检查该部位是否有装备
2. 应用负向属性变化
3. 更新用户实体
4. 重新计算战斗属性

## 5. 装备穿戴流程

### 5.1 穿戴装备

#### 5.1.1 前置校验
1. 用户存在
2. 物品存在于背包中
3. 物品类型为装备
4. 装备等级要求 ≤ 用户等级

#### 5.1.2 穿戴流程
```
接收穿戴请求
    ↓
验证物品存在性和合法性
    ↓
检查部位是否已有装备
    ↓
若有 → 卸下原装备（返回背包）
    ↓
标记新装备为equipped=true
    ↓
计算属性变化（正向）
    ↓
更新用户实体属性
    ↓
保存装备和用户数据
    ↓
返回属性变化详情
```

#### 5.1.3 响应示例
```
成功装备【铁剑】！

【属性变化】
  攻击：+15
  力量：+5
```

### 5.2 卸下装备

#### 5.2.1 前置校验
1. 用户存在
2. 该部位有已装备的装备

#### 5.2.2 卸下流程
```
接收卸下请求
    ↓
验证部位是否有装备
    ↓
标记装备为equipped=false
    ↓
计算属性变化（负向）
    ↓
更新用户实体属性
    ↓
保存装备和用户数据
    ↓
返回属性变化详情
```

#### 5.2.3 响应示例
```
已卸下【胸甲】部位的【皮甲】。

【属性变化】
  防御：-8
  体质：-3
  HP上限：-60
```

## 6. 性能优化

### 6.1 缓存策略

#### 6.1.1 物品模板缓存
- 物品模板变化频率极低
- 使用Spring Cache缓存所有模板
- 缓存Key：templateId

#### 6.1.2 装备属性预计算
- 装备生成时预计算最终属性
- 避免每次查询都重新计算

### 6.2 批量操作

#### 6.2.1 批量添加物品
```java
public void addItems(Long userId, List<ItemStack> items) {
    // 一次性处理多个物品
    // 减少数据库交互次数
}
```

#### 6.2.2 背包分页查询
- 当背包物品过多时，支持分页加载
- 每次加载20格

## 7. 扩展性设计

### 7.1 新装备类型

**步骤**：
1. 在ItemTemplate中添加新type
2. 定义新的slot部位（如"BACK"背部饰品）
3. 在装备穿戴逻辑中添加新部位处理
4. 更新UI显示

### 7.2 新词条系统

**步骤**：
1. 扩展affixes JSONB结构
2. 添加新词条类型（如元素伤害、技能加成）
3. 在战斗系统中应用新词条效果

### 7.3 装备强化系统

**设计思路**：
- 添加forge_level字段（已存在）
- 强化成功率随等级下降
- 强化失败可能降级或损坏
- 消耗材料：强化石、金币
