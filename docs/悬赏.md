# 悬赏系统

## 1. 概述

悬赏是玩家在地图节点接取的限时任务。接取后需等待指定时长，提交后根据预确定的奖励池发放物品/灵石/灵兽卵/装备，期间可能触发 D20 旅行事件。

核心特性：
- **种子预确定奖励**: 接取时即确定奖励内容并写入 DB，提交时直接发放，不重算
- **同日幂等**: 同一用户同一天接取同一悬赏，奖励相同
- **LLM 美化**: 完成时调用 AI 生成叙事描述

## 2. 数据库设计

### 2.1 xt_bounty — 悬赏配置表

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL | PK |
| `map_id` | BIGINT | 所属地图 ID |
| `name` | VARCHAR(100) | 悬赏名称 |
| `description` | TEXT | 悬赏描述 |
| `duration_minutes` | INTEGER | 悬赏耗时（分钟） |
| `rewards` | JSONB | 奖励池（加权随机） |
| `require_level` | INTEGER | 最低接取等级 |
| `event_weight` | INTEGER | 旅行事件触发权重 |

**rewards JSONB 格式**:

```json
[
  {"type": "rare_item", "weight": 50, "minCount": 1, "maxCount": 3, "name": "稀有物品"},
  {"type": "spirit_stones", "weight": 30, "minAmount": 50, "maxAmount": 200, "name": "灵石"},
  {"type": "BEAST_EGG", "weight": 20, "name": "灵兽卵"},
  {"type": "equipment", "weight": 10, "templateId": 5, "name": "装备"}
]
```

- `type`: 奖励类型，支持 `rare_item` / `spirit_stones` / `BEAST_EGG` / `equipment`
- `weight`: 加权随机权重
- `minCount` / `maxCount`: (rare_item) 产出稀有物品数量范围
- `minAmount` / `maxAmount`: (spirit_stones) 灵石数量范围
- `templateId`: (equipment) 指定装备模板 ID

### 2.2 xt_user_bounty — 用户悬赏记录表

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL | PK |
| `user_id` | BIGINT | 用户 ID |
| `bounty_id` | BIGINT | 悬赏 ID |
| `bounty_name` | VARCHAR(100) | 悬赏名称（快照） |
| `start_time` | TIMESTAMP | 接取时间 |
| `duration_minutes` | INTEGER | 悬赏耗时（分钟） |
| `rewards` | JSONB | 预确定的奖励物品 |
| `status` | VARCHAR(16) | `active` / `completed` / `abandoned` |

- `rewards` 在接取时由种子随机计算后写入，提交时直接读取发放
- `status` 状态机: `active` → `completed` / `abandoned`

## 3. 奖励体系

### 3.1 奖励类型

| 类型 | 来源 | 说明 |
|---|---|---|
| `rare_item` | 地图特产 | 加权随机选取 N 个物品 |
| `spirit_stones` | 固定数值 | 直接发放灵石到用户账户 |
| `beast_egg` | ItemTemplate (BEAST_EGG) | 随机选取一颗灵兽卵 |
| `equipment` | 装备模板 | 加权随机选取一件装备 |

### 3.2 物品选取 (rare_item)

先从 `rewards` 加权随机选中 `rare_item` 类型，再从当前地图 `xt_map_node.specialties` 中按 `weight` 加权随机选取 `minCount ~ maxCount` 件。

特产 JSONB 格式:

```json
[
  {"templateId": 1, "weight": 50},
  {"templateId": 3, "weight": 10}
]
```

- `templateId` 为 item_template 主键，`weight` 为权重

## 4. 奖励预确定机制 (种子随机)

接取悬赏时，用确定性的种子构建随机数生成器，预计算奖励内容并写入 `xt_user_bounty.rewards`。

```
seed = userId * 31 + LocalDate.now().toEpochDay()
rng = new Random(seed)
```

**特性**:
- 同用户同日接取同一悬赏 → 奖励相同（幂等）
- 跨日 seed 变化 → 不同日期奖励不同
- 奖励写入 DB 后不再变动，提交时直接读取

### 4.1 预确定流程

```
1. 构建 rng(seed)
2. 加权随机从 bounty.rewards 中选取一条顶层奖励类型 (rare_item/spirit_stones/BEAST_EGG/equipment)
3. 根据类型展开具体物品:
   - rare_item     → 从 mapNode.specialties 加权选取 minCount~maxCount 件
   - spirit_stones → 生成 SpiritStonesReward {_rewardType: "spirit_stones", amount: N}
   - beast_egg     → 从 ItemTemplate(BEAST_EGG) 随机选 1 个
   - equipment     → 从 EquipmentTemplate(templateId) 生成 1 件
4. 写入 xt_user_bounty.rewards (JSONB)
```

### 4.2 内部标记

预确定的 rewards 使用 `_rewardType` 标记字段区分奖励类型（ItemReward 无标记，作为默认兜底）:

```json
[
  {"_rewardType": "spirit_stones", "amount": 100},
  {"_rewardType": "BEAST_EGG", "name": "灵兽卵", "templateId": 3, "quantity": 1},
  {"_rewardType": "equipment", "name": "装备", "templateId": 5, "quantity": 1}
]
```

- `spirit_stones`: `_rewardType = "spirit_stones"`, 字段 `amount`
- `BEAST_EGG`: `_rewardType = "BEAST_EGG"`, 字段 `name` / `templateId` / `quantity`
- `equipment`: `_rewardType = "equipment"`, 字段 `name` / `templateId` / `quantity`
- 物品奖励 (ItemReward): 无 `_rewardType`, 字段 `templateId` / `name` / `quantity`

## 5. 业务流程

### 5.1 操作流程

```
#悬赏列表
  → 查询当前地图的悬赏 (过滤等级)
  → 显示 ID / 名称 / 描述 / 推荐等级

#悬赏接取 [ID]
  → 验证: 用户 IDLE / 悬赏存在 / 地图匹配 / 等级满足
  → 种子预确定奖励 → 写入 xt_user_bounty (status=active)
  → 用户 status → BOUNTY

  ... 等待 duration_minutes 分钟 ...

#悬赏提交
  → 验证: 用户 BOUNTY / 存在 active 记录 / 时间 ≥ duration_minutes
  → 读取预存 rewards → 发放物品 + 灵石 + 装备 + 灵兽卵
  → D20 事件判定 (D20 ≤ 10 触发)
  → LLM 美化叙事
  → 标记 completed → 用户 status → IDLE

#悬赏放弃
  → 验证: 用户 BOUNTY / 存在 active 记录
  → 标记 abandoned (无任何产出) → 用户 status → IDLE
```

### 5.2 状态约束

| 操作 | 要求用户状态 | 完成后状态 |
|---|---|---|
| 悬赏列表 | IDLE | — |
| 悬赏接取 | IDLE | BOUNTY |
| 悬赏提交 | BOUNTY | IDLE |
| 悬赏放弃 | BOUNTY | IDLE |

### 5.3 提交时时间检查

计算从接取到当前时间的已过分钟数，若小于悬赏要求时长则抛出异常提示剩余时间。

---

## 6. 旅行事件

悬赏完成时触发 D20 事件判定，与旅行系统共享同一事件机制。

- D20 ≤ 10 → 触发事件（从 `mapNode.travel_events` 加权随机选取）
- D20 > 10 → 安全通过，无事件

| 事件 | 效果 |
|---|---|
| AMBUSH | 受到 10-29 点伤害 |
| FIND_TREASURE | 获得 5-19 灵石 |
| WEATHER | 纯叙事（"遭遇毒雾天气..."） |
| SAFE_PASSAGE | 无事发生 |

### 7.2 核心组件

**BountyRewardPool**: 奖励池密封接口，4 种变体:
- `RareItem` — 加权从地图特产选取，含 `minCount`/`maxCount` 数量范围
- `SpiritStones` — 灵石奖励，含 `minAmount`/`maxAmount` 数值范围
- `BeastEgg` — 灵兽卵，从 BEAST_EGG 类型 ItemTemplate 随机选取
- `EquipmentReward` — 装备奖励，指定 `templateId` 直接生成

通过 Jackson 多态反序列化，`type` 字段值: `rare_item` / `spirit_stones` / `BEAST_EGG` / `equipment`。

**BountyRewardItem**: 预存奖励密封接口，4 种变体:
- `ItemReward` — 普通物品（无 `_rewardType` 标记）
- `SpiritStonesReward` — 灵石（`_rewardType: "spirit_stones"`）
- `BeastEggReward` — 灵兽卵（`_rewardType: "BEAST_EGG"`）
- `EquipmentRewardItem` — 装备（`_rewardType: "equipment"`）

提供 `toMap()` 序列化为 JSONB 兼容 Map，`parse()` / `parseOne()` 反序列化。

**BountyRepository**: 仓储接口，提供按 ID 查询、按地图 ID 查询列表的功能。

**UserBountyRepository**: 仓储接口，提供按用户 ID 查找 active 悬赏、持久化记录的功能。

**BountyService**: 悬赏业务入口，遵循项目 Service 双层 API 模式。Public 方法标注 `@Authenticated`，接受 `(PlatformType, String openId, ...)` 参数，委托给同名的 Internal 方法。Internal 方法使用 `UserContext.getCurrentUserId()` 获取用户 ID。

主要内部方法职责:
- `listBounties(userId)` — 查询用户所在当前地图的悬赏，按等级过滤后返回 VO 列表
- `startBounty(userId, bountyId)` — 验证用户状态/悬赏存在/地图匹配/等级门槛，种子预计算奖励后写入 DB，更新用户状态
- `getBountyStatus(userId)` — 查询 active 悬赏，计算已过/剩余时间，返回状态 VO
- `abandonBounty(userId)` — 验证状态，标记悬赏为已放弃，恢复用户空闲状态
- `determineRewards()` — 核心预确定方法: 加权随机选择奖励池类型，按类型展开具体奖励物品

**BountyCombatService**: 独立组件，负责悬赏完成子流程:
- `completeBounty(userId)` — 验证状态/时间，读取预存奖励，调用发放逻辑，判定旅行事件，LLM 美化，更新记录和用户状态
- `addRewardsToInventory()` — 区分 ItemReward/BeastEggReward 调用 StackableItemService、EquipmentRewardItem 调用 EquipmentService，SpiritStonesReward 直接更新用户灵石
- `resolveEvent()` — D20 判定，加权随机从地图节点选取事件
- `beautifyBountyCompletion()` — 调用 ExplorationDescriptionFunction 生成叙事描述

## 9. 数据流图

```
接取悬赏:
  Player → QQ 监听器捕获命令
         → 命令处理器解析参数、调用业务服务
         → 服务层验证用户状态/悬赏存在/地图匹配/等级
         → 预计算: 从 bounty.rewards 加权随机选类型
         →    rare_item: 从 mapNode.specialties 加权随机选 N 件
         →    beast_egg: 从 ItemTemplate 随机选 1 个
         →    equipment: 从 EquipmentTemplate 按 templateId 生成
         → 写入 xt_user_bounty (含预确定奖励 JSONB)
         → 用户状态设为 BOUNTY
         → 返回确认文本

提交悬赏:
  Player → QQ 监听器捕获命令
         → 命令处理器调用业务服务
         → 服务层验证用户状态/时间已到
         → 读取 xt_user_bounty.rewards
         → 发放物品/灵兽卵 (StackableItemService)
         → 发放装备 (EquipmentService)
         → 发放灵石 (直接更新用户账户)
         → D20 事件判定 (从 mapNode.travel_events 加权随机)
         → LLM 美化叙事 (ExplorationDescriptionFunction)
         → 标记悬赏为 completed + 用户状态恢复 IDLE
         → 返回 BountyRewardVO (含奖励描述 + 事件描述)
```
