# 师徒系统设计

## 1. 概述

师徒系统是现有护道系统（`DaoProtection`）的上层封装——收徒自动建立护道关系，护道关系也可独立于师徒存在。师傅纯付出、徒弟纯获益，定位为高境界玩家引导新人的社交机制。

### 设计原则

- **护道泛化**：师徒自动建立护道关系，复用现有护道突破加成。护道可独立于师徒存在（不拜师也能找人护道）
- **师傅零物质收益**：师傅不收徒获取灵石/经验/声望，纯社交驱动
- **一徒一师**：每个徒弟同时只有一个师傅
- **师徒同宗**：师徒必须在同一宗门（或同为散修），师傅的宗门归属决定徒弟的宗门归属
- **即时建立**：拜师/收徒无需对方同意，直接建立关系

---

## 2. 关系模型

### 2.1 与护道的关系

```
师徒 = 护道 + 额外徒弟收益（修炼速度加成）

护道可独立存在：
  高阶修士 ←护道→ 低阶修士（仅突破加成）
  
师徒自动包含护道：
  师傅 ←护道+师徒→ 徒弟（突破加成 + 修炼速度加成）
```

### 2.2 数量限制

| 方向 | 上限 | 说明 |
|------|:----:|------|
| 一位师傅最多收徒 | **3 人** | 复用护道 `MAX_PROTECTOR_COUNT` |
| 一位徒弟同时只能有 | **1 人** | 一名师傅 |

### 2.3 建立条件

| 条件 | 说明 |
|------|------|
| 师傅境界 | 至少高于徒弟 **一个大境界**（如筑基可收炼气） |
| 宗门归属 | 双方必须在同一宗门，或同为散修（均不在任何宗门） |
| 冷却检查 | 徒弟的 `cooldown_until` 已过期（被逐出/叛师后 24h 内不可重新拜师） |
| 已有关系 | 徒弟无 ACTIVE 状态的师徒关系；若存在旧记录（DISMISSED/RENEGED 且冷却已过），会自动覆盖更新 |

## 3. 拜师与收徒

### 3.1 拜师流程（即时建立）

```
玩家：拜师 [道号]
  ↓
① 验证目标存在且境界高于自己至少一个大境界
② 验证自己无 ACTIVE 师徒关系（旧冷却记录会被覆盖）
③ 验证目标未满收徒上限（≤ 3 人 ACTIVE）
④ 验证双方在同一宗门或同为散修
⑤ 直接建立师徒关系 + 自动建立护道关系
```

### 3.2 收徒流程（即时建立）

```
玩家：收徒 [道号]
  ↓
① 验证自己未满收徒上限
② 验证目标无 ACTIVE 师徒关系
③ 验证目标境界低于自己至少一个大境界
④ 验证双方在同一宗门或同为散修
⑤ 直接建立师徒关系 + 自动建立护道关系
```

### 3.3 命令

| 指令 | 说明 |
|------|------|
| `拜师 [道号]` | 向目标拜师（直接建立关系） |
| `收徒 [道号]` | 收目标为徒（直接建立关系） |
| `师徒` | 查看师徒关系信息 |
| `逐出师门 [道号]` | 师傅将徒弟逐出师门 |
| `叛师` | 徒弟叛离师门 |

## 4. 徒弟收益

### 4.1 护道突破加成（复用）

师徒关系自动建立护道关系，徒弟突破时师傅提供成功率加成：

```
单人加成 = 5% + (师傅等级 - 徒弟等级) × 1%
总加成上限 = 20%
仅同地点护道者生效
```

### 4.2 修炼速度被动加成

徒弟在挂机修炼时（历练/挂机经验获取）获得额外速度加成：

```
加成倍率 = 1.0 + (师傅等级 - 徒弟等级) × 0.002
上限 = 1.5（最高 50% 加成）
```

**实现方式**：历练结算时检测师徒关系，取师傅当前等级计算加成，不要求同地点。

---

## 5. 师傅收益

**师傅无任何物质收益。** 不收灵石、不获经验、不返贡献。纯社交动机——帮助新人、门派传承感。

---

## 6. 宗门绑定规则

师徒关系与宗门归属紧密绑定，规则如下：

| 场景 | 规则 |
|------|------|
| **建立师徒** | 双方必须在同一宗门，或同为散修 |
| **师傅加入宗门** | 所有 ACTIVE 徒弟自动同步加入同一宗门（忽略徒弟个人意愿） |
| **师傅退出宗门** | 所有 ACTIVE 徒弟自动叛师并退出宗门，受 24 小时冷却惩罚 |
| **师傅被踢出宗门** | 所有 ACTIVE 徒弟自动叛师并退出宗门，受 24 小时冷却惩罚 |
| **师傅解散宗门** | 所有 ACTIVE 徒弟自动叛师并退出宗门，受 24 小时冷却惩罚 |

### 实现要点

- `SectService.inviteMember()` / `applyJoin()`：成员加入后调用 `syncSectForMaster()`，遍历其所有 ACTIVE 徒弟自动加入
- `SectService.leaveSect()` / `kickMember()` / `dismissSect()`：成员离开/被踢后调用 `handleMasterSectLeave()`，遍历其所有 ACTIVE 徒弟逐个叛师
- `MasterApprenticeService.establishMasterApprentice()`：建立前先校验 `isInSameSect()`
- `MasterApprenticeService.isOnCooldown()`：检查 `cooldown_until` 是否已过期，而非永久禁止

## 7. 出师机制

### 7.1 自动出师

徒弟等级达到师傅当前的境界大阶时自动出师：

```
示例：师傅等级 45（化神期）
      徒弟达到等级 41（化神期首级）→ 自动出师
```

### 7.2 出师效果

| 变化 | 说明 |
|------|------|
| 师徒关系 | 状态设为 GRADUATED，记录 graduated_at |
| 护道关系 | 同时解除（突破时不再获得师傅加成） |
| 出师奖励 | **无**（出师即最大回报） |

---

## 8. 关系解除

### 8.1 逐出师门（师傅发起）

- 师傅单方面将徒弟逐出师门
- 状态设为 DISMISSED，设置 `cooldown_until = now + 24h`
- 徒弟自动退出宗门（若在同宗），贡献值清零
- 护道关系同步解除
- 冷却期内徒弟无法重新拜师

### 8.2 叛师（徒弟发起）

- 徒弟单方面叛离师门
- 状态设为 RENEGED，设置 `cooldown_until = now + 24h`
- 徒弟自动退出宗门，贡献值清零
- 护道关系同步解除
- 冷却期内徒弟无法重新拜师

### 8.3 重新建立关系

冷却期过后（`cooldown_until < now`），徒弟可重新拜师/被收徒。`establishMasterApprentice()` 会更新已有记录（覆盖旧状态、旧师傅 ID），而非插入新记录。

## 9. 数据模型

### 9.1 `master_apprentice` — 师徒关系表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGSERIAL | PK |
| `master_id` | BIGINT | 师傅用户 ID |
| `apprentice_id` | BIGINT | 徒弟用户 ID |
| `status` | VARCHAR(16) | ACTIVE / GRADUATED / DISMISSED / RENEGED |
| `created_at` | TIMESTAMP | 建立时间 |
| `graduated_at` | TIMESTAMP | 出师时间（NULL=未出师） |
| `cooldown_until` | TIMESTAMP | 冷却截止时间（DISMISSED/RENEGED 后 24h） |
| `UNIQUE` | (apprentice_id) | 一徒一师 |
| `UNIQUE` | (master_id, apprentice_id) | 防重复 |

### 9.2 与护道表的关联

收徒时双向操作：

```
① 查询 apprentice_id 是否有旧记录 → 有则覆盖更新（复用 ID，重置状态）
② INSERT/UPDATE master_apprentice (master_id, apprentice_id, status='ACTIVE', cooldown_until=NULL)
③ INSERT/获取 xt_dao_protection (protector_id=master, protege_id=apprentice)
   若已有护道关系则复用
```
