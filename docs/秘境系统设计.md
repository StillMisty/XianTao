# 秘境系统设计

## 1. 设计原则

- **文字挂机 MUD**：所有战斗为后台模拟，无实时操作
- **进入无限制，奖励有限制**：可随意进入探索/带人，仅每日前 N 次通关有奖励
- **无定时任务**：秘境生命周期由玩家交互事件驱动
- **纯社交驱动**：老带新无额外系统奖励
- **天地叙事**：秘境不是分层打怪，而是探索"洞天福地"——由外向内递进

## 2. 定位

| 定位 | 说明 |
|------|------|
| **稀有资源产出地** | 专属掉落池，产出外界无法获取的物品（限定装备、技能、材料） |
| **战斗验证场** | 递进难度检验 Build，首通/速通记录 |
| **老带新社交** | 可组队进入，高境界带低境界无额外奖励，纯社交驱动 |

---

## 3. 秘境结构：外围 → 内围 → 核心

### 3.1 区域递进

每个秘境固定三重天地：

```
┌─────────────────────────────────────────┐
│  外围 (Outer Area)                       │
│  ├── 灵药圃  │  试炼碑  │  散修遗居      │  ← 低难度，基础产出
│  └── 通关条件：探索到通道入口              │
├─────────────────────────────────────────┤
│  内围 (Inner Area)                       │
│  ├── 灵兽巢穴 │ 炼器室  │ 残破丹房       │  ← 中难度，区域相性产出
│  │   藏经阁   │ 阵法遗址                  │
│  └── 通关条件：击败区域守护者              │
├─────────────────────────────────────────┤
│  核心 (Core Area)                        │
│  ├── 传承殿   │  宗门宝库  │  镇守BOSS    │  ← 高难度，顶级产出
│  └── 通关条件：击败镇守 BOSS              │
└─────────────────────────────────────────┘
```

每个区域有若干**建筑/遗址（POI）**，每个 POI 是一种可探索的节点，产出与区域相性一致。

### 3.2 区域相性

每个秘境整体有一个**天地属性**（继承自入口地图节点的特产/氛围），区域内建筑产出与该属性绑定：

| 秘境主题 | 外围产出 | 内围产出 | 核心产出 |
|---------|---------|---------|---------|
| 紫府秘境（金） | 基础矿石 | 玄铁装备材料、金属性法决 | 紫府剑诀、金系至宝 |
| 青木秘境（木） | 低阶灵药 | 稀有药材、木属性丹药 | 青木长生诀、木系至宝 |
| 玄水秘境（水） | 灵泉水 | 水属性法决、灵兽蛋 | 玄水真经、水系至宝 |

每个秘境产出的物品只在该秘境的特定区域掉落——想拿金系至宝就必须打到紫府秘境核心。

### 3.3 建筑类型

| POI 类型 | 交互方式 | 产出相性 | 示例 |
|----------|---------|---------|------|
| 灵药圃 | 采集（非战斗） | 灵药、种子，匹配秘境属性 | 外围→散修药圃 |
| 试炼碑 | 战斗 | 经验、基础法决 | 外围→入门试炼碑 |
| 散修遗居 | 搜索（非战斗） | 随机材料、低级装备 | 外围→废弃茅屋 |
| 灵兽巢穴 | 战斗 | 兽卵、兽材 | 内围→青风狼巢穴 |
| 炼器室 | 采集 + 小战斗 | 矿石、装备材料 | 内围→玄铁炼器室 |
| 残破丹房 | 搜索 | 丹方、丹药 | 内围→丹房遗址 |
| 藏经阁 | 搜索 + 禁制战斗 | 法决玉简 | 内围→残破藏经阁 |
| 阵法遗址 | 解谜（战斗或正确操作） | 阵法材料、特殊物品 | 内围→护山阵基 |
| 守护者 | BOSS 战斗 | 区域通关信物 | 内围→妖兽王 |
| 传承殿 | 特殊战斗 | 终极技能/限定装备 | 核心→祖师传承 |
| 宗门宝库 | 采集（需通关后开启） | 顶级装备/大量灵石 | 核心→藏宝室 |
| 镇守 BOSS | 终极战斗 | 首通称号、限定掉落 | 核心→上古凶兽 |

## 4. 推进边界

### 4.1 外围 → 内围

```
条件：探索完当前区域所有 POI（或找到"通道"类 POI）
动作：玩家输入「秘境继续」或自动解锁提示
结果：进入内围，可探索内围建筑
```

### 4.2 内围 → 核心

```
条件：击败内围守护者（BOSS 战斗）
       守护者掉落"核心令符"，队伍中一人持有即可
动作：玩家输入「秘境继续」
结果：进入核心，可挑战最终 BOSS
```

### 4.3 核心通关

```
条件：击败镇守 BOSS
动作：自动完成
结果：秘境通关 → 结算奖励 → 强制退出实例
```

### 4.4 推进归属

**单人推进**：玩家独立推进区域，彼此不影响。组队时，区域状态以队长为准——队员加入时自动同步到队长进度（只能匹配或落后，不能超越）。

**组队推进**：队长负责决策"继续"，所有队员同步推进。队员可以离队，离队后保留当前秘境实例中的进度，可自行继续或退出。

---

## 5. 退出边界

| 退出方式 | 触发 | 奖励 | 说明 |
|---------|------|------|------|
| **主动撤退** | `秘境撤退` | 保留已获奖励 | 随时可退出 |
| **战败退出** | 全队团灭 (玩家+灵兽 HP=0) | 保留已获奖励 | 复活后回到入口地图节点 |
| **通关退出** | 击败核心 BOSS | 全部奖励 + 通关额外奖励 | 通关后自动退出 |
| **超时退出** | 进入秘境后超过 T 小时（如 4h） | 保留已获奖励 | 防止挂机占实例 |

**进入入口**：玩家退出秘境后，回到入口地图节点（HIDDEN_ZONE），可再次进入。

## 6. 数据模型

### 6.1 `dungeon_template` — 秘境模板

```sql
CREATE TABLE dungeon_template (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(64) NOT NULL,        -- "紫府秘境"
    description     TEXT,
    map_node_id     BIGINT NOT NULL REFERENCES xt_map_node(id),  -- 入口地图节点
    element_type    VARCHAR(16),                 -- 秘境属性 (METAL/WOOD/WATER/FIRE/EARTH)
    min_level       INT NOT NULL,                -- 最低境界要求
    max_level       INT NOT NULL,                -- 最高境界要求
    max_team_size   INT NOT NULL DEFAULT 3,
    timeout_hours   INT NOT NULL DEFAULT 4,      -- 超时时间
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 每个秘境的入口在地图节点中，类型为 HIDDEN_ZONE
-- 见 docs/地图旅行历练.md §3.1
```

### 6.2 `dungeon_poi_config` — 建筑配置

```sql
CREATE TABLE dungeon_poi_config (
    id                  BIGSERIAL PRIMARY KEY,
    dungeon_id          BIGINT NOT NULL REFERENCES dungeon_template(id),
    area                VARCHAR(16) NOT NULL,    -- OUTER / INNER / CORE
    name                VARCHAR(64) NOT NULL,    -- "散修遗居" / "青风狼巢穴"
    poi_type            VARCHAR(16) NOT NULL
                        CHECK (poi_type IN ('GATHER', 'COMBAT', 'SEARCH', 'BOSS')),

    -- 建筑交互配置
    monster_pool        JSONB,                   -- [{monsterTemplateId, weight}] 仅 COMBAT/BOSS
    loot_pool           JSONB NOT NULL,           -- [{templateId, weight, minQty, maxQty}]
    affinity_tags       JSONB,                   -- ["HERB", "WOOD"] 产出标签，匹配秘境相性

    -- 推进条件
    is_passage          BOOLEAN NOT NULL DEFAULT FALSE,  -- 是通往下一区的通道
    unlock_condition    VARCHAR(64),              -- 解锁此 POI 的条件（如"需要核心令符"）

    -- 状态耗竭
    is_one_time         BOOLEAN NOT NULL DEFAULT TRUE,   -- 队伍中是否只能探索一次
    exhausted_hours     INT,                     -- 个人探索后冷却时间（NULL=不可重复）

    UNIQUE (dungeon_id, area, name)
);
```

### 6.3 `dungeon_instance` — 运行时实例

```sql
CREATE TABLE dungeon_instance (
    id              BIGSERIAL PRIMARY KEY,
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    leader_id       BIGINT NOT NULL REFERENCES xt_user(id),
    team_id         BIGINT REFERENCES team(id),   -- 关联队伍（多人时），单人时为 NULL
    current_area    VARCHAR(16) NOT NULL DEFAULT 'OUTER',
                    CHECK (current_area IN ('OUTER', 'INNER', 'CORE', 'COMPLETED')),

    -- 推进状态
    passage_unlocked BOOLEAN NOT NULL DEFAULT FALSE,  -- 通往下一区的通道是否已开启
    explored_pois   JSONB NOT NULL DEFAULT '[]'::jsonb,  -- 已探索的 poi_config_id 列表

    -- 生命周期
    status          VARCHAR(16) NOT NULL DEFAULT 'ACTIVE'
                    CHECK (status IN ('ACTIVE', 'COMPLETED', 'FAILED', 'ABANDONED')),
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expires_at      TIMESTAMP NOT NULL,           -- 超时时间
    completed_at    TIMESTAMP,

    UNIQUE (leader_id, dungeon_id, status)  -- 每人每秘境同时只能有一个 ACTIVE 实例
    -- 注意：status='ACTIVE' 时唯一，COMPLETED/FAILED 不冲突
);
```

队伍成员通过关联的 `team` 表的 `team_member` 获取（见 §11），不再维护 `dungeon_member` 表。

### 6.4 `dungeon_progress` — 奖励记录

```sql
CREATE TABLE dungeon_progress (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES xt_user(id),
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    last_reward_date DATE NOT NULL,              -- 最近领奖日期
    reward_count    INT NOT NULL DEFAULT 0,      -- 今日已领奖次数
    daily_limit     INT NOT NULL,                -- 当日上限（境界递增）
    first_clear     BOOLEAN NOT NULL DEFAULT FALSE,
    best_area       VARCHAR(16),                 -- 历史最高到达区域
    UNIQUE (user_id, dungeon_id)
);
```

`daily_limit`：

```java
daily_limit = switch (level) {
    case 0-9   -> 1   // 练气~筑基
    case 10-19 -> 2   // 金丹
    case 20-29 -> 3   // 元婴
    default    -> 4   // 化神+
}
```

### 6.5 `dungeon_first_clear` — 全服首通

```sql
CREATE TABLE dungeon_first_clear (
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    team_members    JSONB NOT NULL,              -- [userId1, userId2, ...]
    clear_time      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    duration_minutes INT,
    UNIQUE (dungeon_id)
);
```

## 7. 秘境生命周期

### 7.1 进入

```
玩家输入：秘境 紫府秘境
         ↓
① 检查玩家所在地图节点 = dungeon_template.map_node_id（HIDDEN_ZONE 类型）
② 检查境界符合 min/max_level
③ 检查无进行中的实例（UNIQUE 约束）
④ 若玩家在队伍中，自动带入全队成员
⑤ 创建 dungeon_instance（current_area=OUTER, passage_unlocked=FALSE, expires_at=now+4h）
⑥ 返回当前区域可探索 POI 列表
```

单人 `秘境 紫府秘境` 直接创建实例进入。若已组队则自动带入全队（见 §7.4）。

### 7.2 探索

```
玩家输入：秘境探索
         ↓
① 查当前区域未探索的 POI 列表（排除已探索 + is_one_time 的 POI）
② 自动选取第一个未探索 POI
   - 若当前区域无剩余 POI 且 passage_unlocked = TRUE → 提示「输入秘境继续进入下一区域」
   - 若当前区域无剩余 POI 且 passage_unlocked = FALSE → 报错（异常状态）
③ 根据 poi_type 执行：
     GATHER → 采集处理（消耗时间，纯产出）
     SEARCH → 搜索处理（消耗时间，概率发现物品/触发战斗）
     COMBAT → 进入战斗（CombatEngine.simulate，BattleScene.DUNGEON）
     BOSS   → 进入战斗（Boss 战，战斗后掉落）
④ 处理掉落（按 loot_pool 加权抽取）
⑤ 更新 explored_pois 加入此 POI
⑥ 检查推进条件（所有非通道 POI 探索完成 → 解锁 passage_unlocked）
```

**战斗 POI** 复用现有 `CombatEngine`，`BattleContext.scene=DUNGEON`，传入 `dungeon_id` 和 `area`。

### 7.3 推进

```
条件：passage_unlocked = TRUE
      外→内：所有外围 POI 已探索
      内→核：已探索内围守护者 POI（BOSS 类型），掉落"核心令符"
      核心→通关：击败镇守 BOSS

玩家输入：秘境继续
         ↓
① current_area 升级（OUTER→INNER→CORE→COMPLETED）
② 如果 current_area = COMPLETED：结算奖励，强制退出
③ 否则：重置 explored_pois，展示新区域建筑列表
```

### 7.4 组队

组队由独立的**组队系统**管理（非秘境专属），详见 §11。

```
队长：组队邀请 张三
   ↓
张三：组队接受
   ↓
（此时张三加入队长的队伍）

队长：秘境 紫府秘境
   ↓
所有队员同步进入第一区域（区域状态以队长已有实例为准，队员自动同步）
```

- 队员加入时，实例区域状态以队长为准
- 队员若有自己的 ACTIVE 实例，需先放弃或等其超时
- 队伍中任何成员可随时 `秘境撤退` 独自退出（不影响其他人）
- 成员全部退出 → 实例 ABANDONED

### 7.5 退出与超时

```
主动退出：秘境撤退
   ↓
① status = ABANDONED
② 发放已获奖励
③ 玩家回到入口地图节点

战败退出：全队团灭
   ↓
① status = FAILED
② 发放战败前已获奖励
③ 玩家回到入口地图节点，恢复至满状态

超时退出：expires_at < now 时玩家下次交互触发
   ↓
① status = ABANDONED
② 发放已获奖励
③ 玩家回到入口地图节点

通关退出：击败核心 BOSS
   ↓
① status = COMPLETED
② 结算全部奖励 + 通关额外奖励
③ 更新 dungeon_progress（reward_count++、best_area、first_clear）
④ 玩家回到入口地图节点
```

## 8. 战斗集成

复用现有 `CombatEngine`：

```java
CombatEngine.simulate(Team, Team, maxRounds, BattleScene.DUNGEON)
```

- `BattleContext` 传入 `scene=DUNGEON` + `dungeonId` + `area`
- 单人队伍 = 玩家 + 出战灵兽
- 多人队伍 = 所有成员 + 各自出战灵兽
- BOSS 的 `MonsterTemplate` 从 `dungeon_poi_config.monster_pool` 加权选取

## 9. 玩家指令

### 9.1 秘境指令

| 指令 | 说明 |
|------|------|
| `秘境` | 查看秘境列表 + 当前进度（有进行中实例时显示区域/已探索 POI/奖励次数） |
| `秘境 {{秘境名}}` | 查看秘境详情（含首通记录）/ 进入秘境（单人） |
| `秘境探索` | 自动探索当前区域下一个未探索 POI（无需指定建筑名，战斗自动结算） |
| `秘境继续` | 推进到下一区域（当前区域所有 POI 已探索且通道开启时可用） |
| `秘境撤退` | 退出秘境并结算 |

**单人流程**：

```
秘境              →  查看可进入秘境列表
秘境 紫府秘境     →  进入外围，看到建筑列表
秘境探索          →  探索下一个 POI（战斗自动结算）
秘境探索          →  探索下一个 POI
秘境探索          →  探索完最后一个 POI → 提示"通道已开启，输入「秘境继续」"
秘境继续          →  进入内围
秘境探索          →  ...
秘境撤退          →  结算退出
```

## 10. 开发流程

1. `domain/dungeon/` — 所有 entity + enum（DungeonArea、PoiType、DungeonStatus）
2. `db/migration/` — 建表 + 种子数据（紫府秘境模板 + POI 配置）
3. `infrastructure/mapper/` + `repository/` — Mapper + RepositoryImpl
4. `service/DungeonService` — 进入/探索/推进/退出/结算，内部 `(Long userId)` + 公开 `@Authenticated`
5. `handle/command/DungeonCommandHandler` — CommandGroup 实现
6. `handle/listener/DungeonListener` — 双平台监听器

## 11. 组队系统

组队由独立的**组队系统**管理，详见 `docs/组队系统设计.md`。秘境通过 `dungeon_instance.team_id` 关联队伍，进入时自动带入全队。队员可随时 `秘境撤退` 独自退出，不影响队友。
