# 秘境系统设计

## 1. 设计原则

- **文字挂机 MUD**：所有战斗为后台模拟，无实时操作
- **进入无限制，奖励有限制**：可随意进入练习/带人，仅每日前 N 次通关有奖励
- **无定时任务**：秘境生命周期由玩家交互事件驱动
- **纯社交驱动**：老带新无额外系统奖励

## 2. 定位

| 定位 | 说明 |
|------|------|
| **稀有资源产出地** | 专属掉落池，产出外界无法获取的物品（限定装备、技能、材料） |
| **战斗验证场** | 多层递进难度，检验 Build 强度，首通/速通记录 |
| **老带新社交** | 可组队进入，高境界带低境界无额外奖励，纯社交驱动 |

## 3. 数据模型

### 3.1 `dungeon_template` — 秘境模板

```sql
CREATE TABLE dungeon_template (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(64) NOT NULL,        -- "紫府秘境"
    description     TEXT,
    map_node_id     BIGINT REFERENCES xt_map_node(id),  -- 入口地图节点
    min_level       INT NOT NULL,                -- 最低境界要求
    max_level       INT NOT NULL,                -- 最高境界要求
    total_floors    INT NOT NULL,                -- 总层数
    entry_cost      BIGINT NOT NULL DEFAULT 0,   -- 入场灵石消耗
    max_team_size   INT NOT NULL DEFAULT 3,
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### 3.2 `dungeon_floor_config` — 每层配置

```sql
CREATE TABLE dungeon_floor_config (
    id              BIGSERIAL PRIMARY KEY,
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    floor_number    INT NOT NULL,                -- 第几层
    monster_pool    JSONB NOT NULL,              -- [{monsterTemplateId: 1, weight: 70}, ...]
    min_monsters    INT NOT NULL DEFAULT 1,
    max_monsters    INT NOT NULL DEFAULT 3,
    description     TEXT,                        -- LLM flavor text
    UNIQUE (dungeon_id, floor_number)
);
```

### 3.3 `dungeon_instance` — 运行时实例

```sql
CREATE TABLE dungeon_instance (
    id              BIGSERIAL PRIMARY KEY,
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    leader_id       BIGINT NOT NULL REFERENCES xt_user(id),
    current_floor   INT NOT NULL DEFAULT 1,
    status          VARCHAR(16) NOT NULL DEFAULT 'ACTIVE'
                    CHECK (status IN ('ACTIVE', 'COMPLETED', 'FAILED', 'ABANDONED')),
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    completed_at    TIMESTAMP,
    UNIQUE (leader_id, dungeon_id, current_floor)  -- 每人每秘境同时只有一个进行中实例
);
```

### 3.4 `dungeon_member` — 队伍成员

```sql
CREATE TABLE dungeon_member (
    id              BIGSERIAL PRIMARY KEY,
    instance_id     BIGINT NOT NULL REFERENCES dungeon_instance(id),
    user_id         BIGINT NOT NULL REFERENCES xt_user(id),
    joined_at       TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 查某玩家的活跃秘境
CREATE INDEX idx_dungeon_member_user
    ON dungeon_member(user_id, instance_id);
```

### 3.5 `dungeon_progress` — 奖励记录

```sql
CREATE TABLE dungeon_progress (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES xt_user(id),
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    last_reward_date DATE NOT NULL,              -- 最近领奖日期
    reward_count    INT NOT NULL DEFAULT 0,      -- 今日已领奖次数
    daily_limit     INT NOT NULL,                -- 当日上限（境界递增）
    first_clear     BOOLEAN NOT NULL DEFAULT FALSE,
    best_floor      INT NOT NULL DEFAULT 0,      -- 历史最高通关层数

    UNIQUE (user_id, dungeon_id)
);
```

`daily_limit` 写入规则（进入秘境时检查并更新）：

```
daily_limit = switch (level) {
    case 0-9   -> 1   // 练气~筑基
    case 10-19 -> 2   // 金丹
    case 20-29 -> 3   // 元婴
    default    -> 4   // 化神+
}
```

如果 `last_reward_date != CURRENT_DATE`，重置 `reward_count = 0`。

## 4. 秘境生命周期

### 4.1 创建与进入

```
玩家输入：秘境 紫府秘境 组队
         ↓
① 查 dungeon_template → 是否存在、境界是否符合
② 查 dungeon_member → 该玩家是否已有进行中的紫府秘境实例（一个同时只能有一个）
③ 创建 dungeon_instance (status=ACTIVE, current_floor=1)
④ 加入 dungeon_member（队长+队员）
⑤ 进秘境 = 进入战斗 → CombatService.simulate() → 结算

若失败（团灭）：status=FAILED→ABANDONED，可重新创建实例再次挑战
若胜利：current_floor++，进入下一层
```

### 4.2 组队

```
队长：秘境 紫府秘境 邀请 @玩家
队员收到通知："队长 邀请你进入 紫府秘境，是否同意？"
   ↓
同意 → 加入 dungeon_member
   ↓
队长：秘境 紫府秘境 开始 → 所有成员进入第一层
```

- 队伍最大 3 人
- 队长创建实例后，队员在进入前可以自由加入/退出
- 一旦开始战斗，不可加人
- 秘境中任何成员退出 → 该成员视为放弃（失去本次奖励资格），不影响其他人继续

### 4.3 层间推进

每通关一层后的选择：

```
你已通过紫府秘境第 3 层，当前状态：
HP: 65/100
灵兽: 满状态

选项：
① 继续挑战第 4 层（推荐等级: 金丹中期）
② 撤退结算（保留已获得奖励）
```

- 撤退 → `status=COMPLETED`，发放已过层数奖励
- 继续 → 进入下一层战斗

不提供层内恢复——阵容管理（丹药、灵兽状态）是战斗验证的一部分。

### 4.4 结算与奖励

```
通关第 N 层
   ↓
if reward_count < daily_limit:
  // 每层固定奖励
  灵石 = N × 100
  随机物品 = 从 dungeon_template 的专属掉落池加权

  // 首通额外
  if !first_clear:
    首通奖励 = 限定物品
    first_clear = true

  reward_count++
  best_floor = MAX(best_floor, N)
else:
  "今日奖励已达上限，下次再来吧"
```

### 4.5 老带新

```
组队秘境难度 = avg(队员境界) 向上取整

奖励结算：
  - 所有队友按个人境界独立计算 daily_limit，互不影响
  - 带队者无额外奖励
  - 低境界队员获得的掉落不受队友境界影响
```

没有"带队收益减半"或"带队加成"。大佬愿意带萌新纯粹是社交行为。低境界者可以提前体验高难度内容，高境界者获得带人的成就感。

## 5. 战斗集成

复用现有 `CombatEngine`：

```
CombatEngine.simulate(Team, Team, 50, BattleScene.DUNGEON)
```

`BattleContext` 传入 `scene=DUNGEON` + `floorNumber`，战斗引擎可根据场景做差异化（未来扩展）。

队伍构建：

```java
// 单人
Team playerTeam = combatService.buildPlayerTeam(userId);
// 可选：带灵兽、嗑丹药备战

// 多人
Team playerTeam = new Team();
for (member : dungeonMembers) {
    playerTeam.add(combatService.buildPlayerTeam(member.userId).getCombatants());
}
```

怪物由 `dungeon_floor_config.monster_pool` 加权选择，传入 `MonsterTemplate` + 境界缩放。

## 6. 首通与速通记录

`dungeon_progress` 已记录 `best_floor` 和 `first_clear`，可用于展示。

可以加一个简单的**全服首通**：

```sql
CREATE TABLE dungeon_first_clear (
    dungeon_id      BIGINT NOT NULL REFERENCES dungeon_template(id),
    team_members    JSONB NOT NULL,              -- [userId1, userId2, ...]
    clear_time      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    duration_minutes INT,
    UNIQUE (dungeon_id)
);
```

全服首通队伍的名字永久刻在秘境入口——社交驱动 + 成就感，零成本。

## 7. 开发流程

1. `domain/dungeon/` — `DungeonTemplate`、`DungeonFloorConfig`、`DungeonInstance`、`DungeonMember`、`DungeonProgress` entities + `DungeonStatus` 枚举
2. `db/migration/` — 建表 + 种子数据（紫府秘境等）
3. `infrastructure/mapper/` — Mapper 接口
4. `infrastructure/repository/` — RepositoryImpl
5. `service/DungeonService` — 创建/进入/推进/结算，内部 `(Long userId)` + 公开 `@Authenticated` 返回 `ServiceResult`
6. `handle/command/DungeonCommand` — `dungeon list`、`dungeon enter`、`dungeon invite` 等
7. `handle/listener/DungeonListener` — 双平台监听器
