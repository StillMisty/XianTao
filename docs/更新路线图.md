## Phase 4 — 福地系统重设计

根据 [福地.md](docs/福地.md) 和 [灵兽.md](docs/灵兽.md) 重构现有代码。

### 4.1 DB Migration

| 迁移文件 | 说明 |
|---------|------|
| `V?.?.?` | `xt_fudi` 删除 `aura_current`, `aura_max`, `last_aura_update` 列 |
| `V?.?.?` | `xt_beast` 删除 `lifespan_days` 列；`recovery_until` 语义改为灵石恢复（非时间等待） |

### 4.2 Domain

#### 移除

| 文件 | 说明 |
|------|------|
| `domain/land/entity/Fudi.java` | 删除 `auraCurrent`, `auraMax`, `lastAuraUpdate` 字段及关联方法 |
| `domain/land/enums/CellType.java` | 删除 `NODE` 枚举值 |
| `domain/land/vo/BeastProduceVO.java` | 合并到收取统一逻辑 |

#### 修改

| 文件 | 说明 |
|------|------|
| `domain/beast/entity/Beast.java` | 删除 `lifespanDays` 字段；`recoveryUntil` 改为灵石恢复触发 |
| `domain/land/entity/Fudi.java` | `gridLayout` 保持 JSONB，移除阵眼相关字段 |

### 4.3 Service

#### FudiService 改动

| 方法 | 说明 |
|------|------|
| 删除 `sacrificeItem*()` 系列 | 献祭系统移除 |
| 删除灵气消耗/恢复逻辑 | 替换为灵石扣减（全局货币） |
| 删除阵眼相关逻辑 | 天劫防御公式：`总防 = Σ(出战灵兽战力 × 护主乘数) + STR × 10 + 劫数 × 50` |
| 修改 `tribulationWin()` | 奖励改为灵石（+连胜×100），删除灵气上限增长 |
| 修改 `harvestCrop()` | 消耗地灵精力 5 点（确认） |
| 修改 `collectProduce()` | 与 harvestCrop 合并为统一收取接口 |
| 修改 `deployBeast()` | 出战上限 2 只，出战不产出 |
| 修改 HP 恢复逻辑 | 消耗灵石瞬间回满，非时间等待 |
| 修改 `evolveBeast()` | 删除进化失败寿命减半惩罚 |
| 修改 `releaseBeast()` | 删除寿命相关产出（善终精华） |

### 4.4 Handle

| 文件 | 说明 |
|------|------|
| `handle/command/FudiCommandHandler.java` | 删除献祭命令；删除阵眼建造选项；合并收获→收取 |
| `handle/onebotv11/FudiHandle.java` | 同步删除献祭/阵眼相关监听 |

| 变动 | 说明 |
|------|------|

| 新增 | `#灵兽出战` / `#灵兽召回` / `#灵兽恢复` |

## 实施顺序依赖

```
Phase 3 ─── 依赖 Phase 2 的 xt_skill + xt_player_skill + SKILL_JADE ItemType
              （SkillService 读写 xt_player_skill，消耗 SKILL_JADE 物品）

Phase 4 ─── 依赖现有 xt_fudi + xt_beast 表结构，进行字段变更和逻辑重构
              （删除灵气/阵眼/献祭/寿命，替换为灵石经济 + 新灵兽机制）

Phase 5 ─── 依赖 Phase 3 的法决 + 战斗系统数据输出
```
