## 🟡 第 3 阶段：架构优化

### 9. 异常控制流反模式
- **模式：** 内部方法抛 `IllegalStateException` → public API `catch` → 转 `ServiceResult.Failure`
- **问题：** 异常开销大（栈快照），语义错位（业务失败 ≠ 程序异常）
- **方案：** 内部方法直接返回 `ServiceResult<T>`，消除 `try-catch`

### 10. 统一 AuthResult 与 ServiceResult
- **当前：** `AuthenticationService.AuthResult`（record）vs `ServiceResult<T>`（sealed interface）—— 功能相同，类型不同
- **每个 public 方法都重复：**
  ```java
  var auth = authService.authenticateAndValidateUser(...);
  if (!auth.authenticated()) return new ServiceResult.Failure<>(auth.errorMessage());
  ```
- **方案：** 提取 AOP 切面统一认证

### 11. ServiceResult.Failure 添加错误码
- **当前：** `record Failure<T>(String errorMessage)` — 仅文本
- **方案：** 增加 `String errorCode`（如 `AUTH_FAILED`、`INSUFFICIENT_FUNDS`），让命令处理器可做差异化 UI 展示

### 12. 全量加载 → 数据库分页
| 文件:方法 | 问题 |
|---|---|
| `ItemUseService.useItem()` | 查全量背包再内存模糊匹配 |
| `StackableItemService.searchStackableItemsByTags()` | 每次标签查询走全量 + stream |
| `StackableItemService.searchStackableItemsByType()` | 同上 |
| `InventoryService.getInventory()` | 全量装备 + 全量物品 |

- **方案：** 在 Repository 层加 `findByUserIdAndType(userId, type, offset, limit)` 方法，用 SQL 的 `LIMIT/OFFSET`
- 模糊匹配移到 SQL 层 `WHERE name ILIKE '%keyword%'`

### 13. `Map<String, Object>` 返回 → 强类型 VO
- **涉及方法：** `deployBeast()`, `releaseBeast()`, `recoverBeast()`, `buildCell()`, `removeCell()`, `upgradeCell()`, `collect()`, `giveGift()`, `getCellStatus()`
- **方案：** 定义对应 VO record（如 `DeployBeastVO`），编译期类型安全

### 14. 补充单元测试（优先模块）
| 模块 | 优先级 | 理由 |
|---|---|---|
| `DefaultCombatEngine` | 高 | 战斗核心逻辑，纯函数可测 |
| `CultivationService.attemptBreakthrough()` | 高 | 突破概率计算、HP/经验/护道联动 |
| `BeastService` 孵化/进化 | 中 | 多分支判定（品质、变异、成功率） |
| `EquipmentService.createEquipment()` | 中 | 随机词缀生成逻辑 |
| `FudiEventGenerator.generateEvents()` | 中 | 时间相关事件生成 |
| `DropProcessor.processMonsterDrops()` | 低 | 掉落权重 |

## 🟢 第 4 阶段：长期改进

### 17. 魔法数字提取常量
```java
// BeastService.java
200   → PRODUCTION_HP_PER_TIER
100   → BASE_HP_MULTIPLIER
0.15  → LEVEL_SPEED_BONUS_PER_LEVEL

// FudiService.java
200, 400, 800, 1600 → UPGRADE_COST_BY_LEVEL

// TrainingService.java
2     → BASE_EXP_PER_MINUTE（已提取 ✓）
5     → MIN_TRAINING_MINUTES
```

### 18. 实体 ActiveRecord 反模式
- **文件：** `User extends Model<User>` 等
- **问题：** MyBatis-Flex `Model` 让实体带 CRUD，破坏 DDD 分层
- **方案：** 长期考虑只继承独立 POJO，CRUD 全部走 Repository

### 19. AI 调用增加熔断/超时
- **文件：** `SpiritChatService.java:86`
- **问题：** `spiritChatClient.prompt().call()` 直接调用，无超时/重试
- **方案：** 配置 OpenAI starter 的 `spring.ai.retry`、或加 Resilience4j `@CircuitBreaker`，避免 AI 故障阻塞 QQ 消息

### 20. 添加 Rate Limiting
- **方案：** 在 Controller/Listener 层加 `@RateLimiter`（Guava RateLimiter 或 Bucket4j），每个用户每种命令设置合理频率上限

### 21. 实现 `handle/web/` REST API
- **当前：** 目录为空（AGENTS.md 已规划但未实现）
- **方案：** 优先暴露：福地状态查询、背包查询、排行榜等只读接口

### 22. 消除重复代码
- `BeastService.hatchBeast()` 与 `hatchBeastWithTemplate()` → 提取 `doHatch(userId, cellId, template)` 公共方法
- `SpiritTools` 的 `collectProduce` / `harvestCrop` → 已合并到第 4 点

### 23. JSONB 安全解析
- **问题：** 多处 unchecked cast（如 `(Map<String, Object>) dropTable.get("equipment")`）
- **方案：** 在 `PgJsonbTypeHandler` 中增加类型校验，或使用 `@SuppressWarnings` + `instanceof` 防御
