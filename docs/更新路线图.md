## 二、P1 — 高风险问题（本迭代修复）

### 2.2 `UserBounty.status` 使用原始 String 而非枚举

**文件**: `domain/bounty/entity/UserBounty.java:33,41-47`

```java
@Column(onInsertValue = "'active'")
private String status;  // "active" / "completed" 是魔法字符串
```

`isActive()` 和 `isCompleted()` 直接硬编码字符串比较。应定义为 `enum BountyStatus { ACTIVE, COMPLETED }`。

### 2.3 枚举 `fromCode()` 三种不同错误处理策略

| 行为 | 涉及的枚举 |
|------|-----------|
| **返回 `null`** | `PlatformType`, `UserStatus`, `AttributeType`, `ItemType`, `EquipmentSlot`, `WeaponType`, `AffixType`, `MonsterType`, `ElementType`, `TravelEventType` |
| **抛出 `IllegalArgumentException`** | `CellType`, `MBTIPersonality`, `MutationTrait`, `EmotionState`, `BeastQuality` |
| **返回默认值** | `Rarity`(COMMON), `SkillType`(ACTIVE), `PillQuality`(NORMAL), `EffectType`(DAMAGE), `BindingType`(NONE) |

调用者必须根据不同的枚举使用不同的错误处理方式。统一为一种策略。

### 2.4 Repository 三种不同的 save 策略

| 模式 | 使用的 Repo | 问题 |
|------|-----------|------|
| `insertOrUpdateSelective` | User, FudiCell, Fudi, Spirit | 仅更新非 null 字段，不会覆盖已设置值 |
| `insertOrUpdate`（非 selective） | StackableItem, Equipment | 覆盖所有列为 null |
| 手动 null-check → `insert`/`update` | Beast, PlayerSkill, MonsterTemplate | 非原子操作 |

三种策略混用会导致不同的行为 Bug。统一为一种。

### 3.9 `DamageCalculator.evaluateExpression` 公式求值脆弱

**文件**: `service/combat/DamageCalculator.java:88-111`

自定义公式求值器仅支持 `+`, `-`, `*`，无括号、无运算符优先级。`2*3+4` 的计算顺序取决于字符串分割顺序。当前公式侥幸正确但脆弱。

### 3.10 Entity 的 `staticConstructor` 与 ORM 冲突风险

**文件**: `User.java:22`, `DaoProtection.java:17`, `Fudi.java:17`, `Spirit.java:18`, `SpiritForm.java:18`, `MapNode.java:18`

```java

```

将无参构造函数设为 `private`。MyBatis-Flex 通过 `setAccessible(true)` 绕过，但如果框架升级构造器解析行为，这些实体可能静默破裂。

### 3.11 `UserBountyRepository` 暴露持久化细节

**文件**: `domain/bounty/repository/UserBountyRepository.java:10-12`

```java

```

领域层的 Repository 接口不应区分 insert 和 update。应为 `void save(UserBounty bounty)` 统一处理。

### 3.12 领域实体贫血化 — 34% 的实体无行为方法

```java
try {
    return CellType.valueOf(code.toUpperCase());

    return null;  // 静默错误

```

当 `cellType` 为 null 时默认返回 `EmptyConfig.class`——DB 读取错误被完全掩盖。

### 4.3 MyBatis JSONB TypeHandler 无 ObjectMapper 配置

**文件**: `JsonbTypeHandler`, `JsonbCollectionTypeHandler`, `CellConfigTypeHandler`, `BountyRewardListTypeHandler`

全部创建 `new ObjectMapper()` 默认配置，无 `JavaTimeModule` 注册。如果任何 JSONB 字段包含 `LocalDateTime` 或 `Instant`，反序列化将失败。

所有 9 个 CommandHandler 都实现 `CommandGroup` 用于自动发现，唯独 HelpCommandHandler 不实现——命令文档中不会自描述。

### 4.5 LeaderboardCommandHandler 缺少错误前缀

### 4.6 `MBTIPersonality` 枚举包含大量硬编码提示词

16 个枚举值，每个包含 100-400+ 字符的系统提示描述。应移至配置文件或数据库。

### 4.7 SpiritPromptTemplates 文本块缩进注入 LLM 提示词

**文件**: `service/ai/SpiritPromptTemplates.java:36-66`

Java 文本块的缩进空白被原样发送给 LLM——浪费 Token 且可能混淆模型。应使用 `.stripIndent()`。

### 4.8 `@Data` Entity 的 equals/hashCode 行为不一致

所有未扩展 `Model<T>` 的 Entity（`Beast`, `Equipment`, `StackableItem`, `FudiCell` 等）的 `equals`/`hashCode` 基于所有字段。实体 null `id` 时基于值，有 `id` 后基于标识——经典的 ORM 反模式。

**文件**: `db/migration/V1.0.1__create_xt_map_node.sql:37-40`

`travel_events` 和 `neighbors` 列的 GIN 索引——这两列很少被用作查询条件。

## 五、架构改进建议（非阻塞，长期规划）

### 5.1 引入应用层 Application Service 门面

当前 Handle 层直接调用多个 Domain Service，缺乏用例编排。引入 Application Service 可：
- 将跨多个 Service 的复杂操作（如战斗后领取奖励+经验+掉落）封装为单一用例
- 减少 Handle 层的业务逻辑泄漏
- 提供清晰的边界用于事务管理

### 5.2 统一 `@Transactional` 策略

目前 `@Transactional` 标注位置不一致（Service 方法 vs Repository 方法）。应：
- 仅在 Application Service 层标注 `@Transactional`
- Domain Service 内部方法不管理事务
- Repository 方法不管理事务（由上层控制）

### 5.3 提取共享内核

`calculateSingleProtectorBonus`, `isInSameLocation`, `getCropTier` 等被多处使用的逻辑应提取至 `top.stillmisty.xiantao.domain.shared` 包。

### 5.4 ItemProperties 深度嵌套重构

`ItemProperties.java`(75 行) 中 `BeastEgg.BeastSkillPool.InnateSkill` 是 3 层嵌套。考虑将每个 Property 类型提取至独立文件，使用包内可见性控制访问。

### 5.5 Spring AI 集成测试

Spring AI 2 的 ChatClient 需要网络连接才能测试。应使用 WireMock 或 Spring AI 的测试支持来验证 Function Calling + 提示词模板。
